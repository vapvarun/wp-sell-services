## Executive summary

WP Sell Services is being extended from a single-vertical freelance marketplace into a four-type **service domain** platform. One CPT, one cart, one order table, one commission engine, one audit log — four distinct "service types" sharing the infrastructure and each contributing its own behavior via a strategy pattern.

**Four service types:**

| Type | What it is | Where | Release |
|---|---|---|---|
| **Standard Service** | Freelance-style work delivered digitally (current behavior wrapped in the new architecture) | Free | 1.2.0 |
| **Digital Download** | Downloadable files sold with signed-URL delivery and license terms | Pro | 1.2.0 |
| **Booking** | Time-slot session reservations (coaching, consulting, tutoring, therapy, advisory) | Pro | 1.3.0 |
| **Rental** | AirBnB-style space reservations (rooms, studios, venues, meeting rooms, workshops, event halls) | Pro | 1.4.0 |

**Explicitly out of scope (not a physical-goods marketplace):**

- No shipping calculation, no tracking numbers, no carrier integration
- No physical product inventory in the e-commerce sense
- No stock management, warehousing, drop-shipping, fulfillment
- No `PhysicalServiceType`
- Strictly the service domain — every type represents an **intangible deliverable or facilitated access**

**Business model:** single Pro SKU. Free ships the foundation + `StandardServiceType`. Pro adds `DownloadServiceType` + `BookingServiceType` + `RentalServiceType`. Customers upgrade once and get every current and future type.

Every decision below is settled. No more architecture debate — implementation starts from these answers.

| # | Question | Answer |

| 2 | Base class with sensible defaults | **Yes** — `AbstractServiceType` implements interface with defaults matching standard-service behavior. Subclasses override only what differs |
| 3 | Field definitions format | **PHP arrays** returned from interface methods (implicit through questions 4/5) |
| 4 | Package meta storage | **JSON blob** inside the existing `_wpss_packages` post meta — atomic, safe against package reindexing, no new table |
| 5 | Service-level settings storage | **Single JSON post meta** key `_wpss_service_type_settings` — consistent with Q4, atomic saves |

| 7 | Download conversation / seller messaging | **Conversation created with `purpose = 'support'`** — buyer can reach out post-purchase, vendor responds. No status-change system messages, auto-closes after 90 days inactivity |

#### Class hierarchy

```
ServiceTypeInterface (interface)
  └── AbstractServiceType (abstract, Free)
        ├── StandardServiceType            (Free)
        ├── DownloadServiceType            (Pro 1.2.0)
        └── AbstractTimePeriodServiceType  (abstract, Pro 1.3.0)
              ├── BookingServiceType       (Pro 1.3.0)
              └── RentalServiceType        (Pro 1.4.0)
```

`AbstractTimePeriodServiceType` centralizes:
- Availability management (delegated to type-specific storage)
- Reservation lifecycle (cart → payment → commit OR cart → expire → release)
- Period collision detection
- ICS calendar file generation with timezone-correct VTIMEZONE blocks
- Reminder scheduling via Action Scheduler
- Reschedule operation with Action Scheduler event unschedule + re-enqueue

Bookings specialize for time slots (9:00-10:00); rentals specialize for date ranges (Mar 15 - Mar 20). Same abstraction, different period granularity.

#### Registry

```php
namespace WPSellServices\ServiceTypes;

class TypeRegistry {
    /** @var array<string, ServiceTypeInterface> */
    private static array $types = array();

    public static function init(): void {
        self::register( new StandardServiceType() );

        $filtered = apply_filters( 'wpss_service_types', self::$types );

        foreach ( $filtered as $type ) {
            if ( $type instanceof ServiceTypeInterface ) {
                self::$types[ $type->get_id() ] = $type;
            }
        }
    }

    public static function register( ServiceTypeInterface $type ): void {
        self::$types[ $type->get_id() ] = $type;
    }

    public static function get( string $id ): ?ServiceTypeInterface {
        return self::$types[ $id ] ?? null;
    }

    public static function get_for_service( int $service_id ): ServiceTypeInterface {
        $type_id = get_post_meta( $service_id, '_wpss_service_type', true ) ?: 'standard';
        return self::get( $type_id ) ?? self::get( 'standard' );
    }

    public static function get_for_order( int $order_id ): ServiceTypeInterface {
        global $wpdb;
        $type_id = $wpdb->get_var( $wpdb->prepare(
            "SELECT service_type FROM {$wpdb->prefix}wpss_orders WHERE id = %d",
            $order_id
        ) ) ?: 'standard';
        return self::get( $type_id ) ?? self::get( 'standard' );
    }

    public static function all(): array {
        return self::$types;
    }

    public static function public_types(): array {
        return array_filter( self::$types, static fn( $t ) => ! $t->is_premium() );
    }
}
```

### Layer 2 — Database schema

#### Storage architecture (three tiers)

The plan uses a **three-tier storage model**. Every piece of data lands in exactly one tier based on its access pattern and query shape:

**Tier 1 — Post types and postmeta (WordPress native)**

Used for: the public-facing "listing" — the thing a buyer discovers, browses, and links to.

- `wpss_service` CPT in `wp_posts` — every service, download, booking, and rental is a post of this type
- `wpss_request` CPT in `wp_posts` — buyer requests
- `_wpss_service_type` post meta — type discriminator (`standard` / `download` / `booking` / `rental`)
- `_wpss_packages` post meta — package array with type-specific fields nested inside (Q4 JSON blob decision)
- `_wpss_service_type_settings` post meta — service-level type config (Q5 JSON meta decision)

**Why post types:** native permalinks (`/services/my-studio/`), search integration, SEO plugin compatibility, theme integration, page builder support, drafts/scheduling/trash for free, proven at WordPress scale.

**Tier 2 — Custom transactional tables**

Used for: high-volume transactional data with indexed hot-path queries.

- `wp_wpss_orders` — every order regardless of type (single source of truth for commission, payouts, refunds, reporting)
- `wp_wpss_conversations` + `wp_wpss_messages` — conversation threads
- `wp_wpss_deliveries` — delivery uploads
- `wp_wpss_proposals` — proposals on buyer requests
- `wp_wpss_audit_log` (1.1.0) — forensic trail
- `wp_wpss_wallet_transactions` + `wp_wpss_withdrawals` + `wp_wpss_vendor_profiles` — money movement
- `wp_wpss_reviews` / `wp_wpss_disputes` / `wp_wpss_order_requirements` — order-scoped state

**Why custom tables:** composite indexes on hot filter columns, faster than `postmeta` scans at scale, clean schema with typed columns, no WordPress post overhead for data that has no permalink or SEO value.

**Tier 3 — Type-specific sibling tables (Pro)**

Used for: type-specific per-order state that doesn't fit the generic order schema.

- `wp_wpss_pro_download_files` + `wp_wpss_pro_download_grants`
- `wp_wpss_pro_booking_slots` + `wp_wpss_pro_booking_reservations` + `wp_wpss_pro_bookings`
- `wp_wpss_pro_rental_availability` + `wp_wpss_pro_rental_reservations` + `wp_wpss_pro_rentals`

**Why sibling tables:** each type has unique state (booking time slots, download grants, rental periods) that doesn't apply to other types. Denormalizing into `wp_wpss_orders` would bloat every row with mostly-null columns. Sibling tables with `order_id` as a foreign key keep the orders table lean while type-specific data stays indexable.

**The one rule connecting all tiers:**

> **Every order has exactly one row in `wp_wpss_orders` regardless of type.** Type-specific data in sibling tables references back via `order_id`. The orders table is the single source of truth for commission, wallet, payout, refund, dispute, and reporting flows.

#### On the `service_type` column choice

The `wp_wpss_orders` table already has a `meta LONGTEXT` JSON column used for ad-hoc per-order data (tax breakdowns, package snapshots, proposal snapshots). A natural question: why add a new `service_type` column when the JSON meta already exists?

**Answer: `service_type` is a hot filter column, not ad-hoc metadata.**

Every dashboard query in the platform filters by it:
- "Show me all my download orders"
- "All bookings for vendor X this week"
- "Revenue by type this quarter"
- "Top-performing services by type"

With a dedicated column + compound index `(service_type, status, created_at)`:
- Every filter is an indexed lookup
- Composite indexes enable efficient multi-column queries
- Zero JSON parsing overhead
- Standard SQL aggregations work natively

With JSON meta:
- `JSON_UNQUOTE(JSON_EXTRACT(meta, '$.service_type'))` in every WHERE clause
- Can't participate in B-tree compound indexes on typical MySQL versions
- Forces full table scans at scale
- 5ms query becomes 500ms query at 100K orders

**The JSON `meta` column continues to exist unchanged** — it still stores tax details, package snapshots, and other ad-hoc per-order data. The `service_type` column is added alongside it as a first-class indexable filter, matching the existing pattern of `platform`, `payment_method`, `status`, and `payment_status` which are all first-class columns for the same reason.

**Three-tier storage summary within the orders table:**

| What | Where | Why |
|---|---|---|
| Hot filter columns (status, vendor_id, service_type, platform, payment_method) | Dedicated indexed columns on `wp_wpss_orders` | Composite indexes, fast lookups |
| Ad-hoc per-order metadata (tax, snapshots, notes) | Existing `meta` JSON column on `wp_wpss_orders` | Schema-less flexibility, no hot filtering needed |
| Type-specific per-order state (bookings, rentals, grants) | Dedicated sibling tables with `order_id` FK | Typed columns, indexed, normalized |

```php
public const STATUS_IN_USE    = 'in_use';     // rentals — buyer has access (checked in)
```

Both constants live in Free (even though only Pro uses them) so audit queries, reports, and cron jobs stay type-agnostic.

**Migration steps:**

1. Bump `SchemaManager::DB_VERSION` from `1.4.0` to `1.5.0`
2. `ALTER TABLE wpss_orders ADD COLUMN service_type`
3. `ALTER TABLE wpss_orders ADD INDEX idx_service_type`
4. `ALTER TABLE wpss_proposals ADD COLUMN request_type`
5. `ALTER TABLE wpss_conversations ADD COLUMN purpose`
6. `ALTER TABLE wpss_conversations ADD INDEX idx_purpose`
7. Backfill `_wpss_service_type = 'standard'` for every existing `wpss_service` post
8. Backfill `wpss_orders.service_type = 'standard'` for every existing row
9. Backfill `wpss_conversations.purpose = 'delivery'` for every existing row
10. Write audit log entry `schema.migration` per step via the 1.1.0 `AuditLogService`

All migrations run via `dbDelta` or batched prepared statements, fully idempotent.

### Layer 3 — State machines

**Standard (unchanged):**

```
pending_payment → pending_requirements → in_progress → pending_approval → completed
                                    ↓               ↓                ↓
                                on_hold        delivered       revision_requested
                                    ↓               ↓                ↓
                               cancelled       (revisions)     (back to in_progress)
```

**Download:**

```
pending_payment → completed  (auto, on payment confirm)
              ↓
          cancelled
```

Auto-complete via `DownloadServiceType::auto_completes_on_payment() = true`. Grants created for every file in the package. Signed-URL delivery via `DownloadsController::get_file_token()`.

**Booking:**

```
pending_payment → scheduled → in_progress → completed
                    ↓             ↓
                cancelled      (cron auto-complete via Action Scheduler)
```

Reschedule is a data-only operation on `wp_wpss_pro_bookings` — old AS events unscheduled, new ones enqueued, no state transition. `STATUS_SCHEDULED` is new.

**Rental:**

```
pending_payment → pending_requirements → in_use → delivered → completed
(label:          (label:               (label:  (label:      (label:
 "Payment         "Awaiting              "Checked "Checked     "Stay
  pending")        check-in")             in")    out")        complete")
                          ↓        ↓          ↓           ↓
                      cancelled  disputed  disputed   disputed
```

`STATUS_IN_USE` is new. Other transitions reuse existing constants with `RentalServiceType::get_status_label()` overrides for display.

**Transitions added per type:**

Each type extends the base state machine via the existing `wpss_order_status_transitions` filter. No hardcoded changes in `OrderService::can_transition()`. Free plugin doesn't know about booking or rental transitions.

```php
        // Rental
        'check_in_utc'   => '2026-04-20 15:00:00',
        'check_out_utc'  => '2026-04-25 11:00:00',

        // Download — no special meta; files resolved from service/package
```

**Order creation flow (`StandaloneOrderProvider::create_orders_from_cart`):**

For each cart item:

1. `$type = TypeRegistry::get_for_service( $item['service_id'] )`
2. `$type->on_order_created( $order_id, $order_data )` — creates type-specific rows (booking, rental, download grants)
3. Initial status = `$type->get_initial_status()`
4. If `$type->auto_completes_on_payment()`, advance to `completed` as soon as payment confirms
5. If `$type->needs_conversation()`, create conversation with `purpose = $type->get_conversation_purpose()`
6. Fire `wpss_order_created` action
7. For time-period types: Action Scheduler enqueues reminder + auto-start + auto-complete events
8. For downloads: create grant rows for every file in the package

**Cart mixing:** allowed. Each item dispatches independently. One gateway charge, multiple orders — one per cart item. Bundled receipt email listing all orders, plus per-type emails per order.

**Booking/rental reservation lifecycle:**

1. `POST /cart/add` with `slot_start/end` or `check_in/out` — creates reservation row with 15-minute TTL
2. Countdown timer in cart UI
3. `POST /cart/checkout` validates reservation still valid; if expired → checkout blocks with item-specific error
4. Payment success → reservation converted to permanent booking/rental row, reservation deleted
5. Payment failure → reservation left in place to expire naturally OR immediately released
6. Cart abandonment → reservation expires on its own via Action Scheduler event scheduled at `expires_at`

#### REST API conventions (wp-plugin-development compliance)

Every new REST endpoint MUST follow these rules — they are non-negotiable and enforced via CI + code review:

**1. Namespace and verbs:**
- Namespace: `wpss/v1` (consistent with existing)
- `GET` reads, `POST` creates, `PATCH` updates, `DELETE` deletes
- Never overload POST for reads or updates

**2. Permission callbacks:**
- Return `WP_Error` with HTTP status 401 (unauthenticated) or 403 (forbidden), **never bare `false`**
- `check_permissions()` from base `RestController` handles the common cases
- Write operations use `check_admin_permissions()` for admin-only routes
- Type-specific caps enforced via `$type->required_vendor_caps()` inside the permission callback

**3. Permission map is filterable:**
```php
/**
 * Filter the permission map for WP Sell Services REST endpoints.
 *
 * @param array  $caps      Array of capability names required for this endpoint.
 * @param string $route     Route identifier (e.g. 'bookings.create').
 * @param mixed  $object    Object being acted on (null for collection endpoints).
 */
$caps = apply_filters( 'wpss_permission_map', $caps, $route, $object );
```

Sites can override per-route caps via this filter without core changes.

**4. Response shape filters:**

Every endpoint response runs through a type-specific filter before being returned:

```php
$data = apply_filters( 'wpss_rest_prepare_order', $data, $order, $request );
$data = apply_filters( 'wpss_rest_prepare_booking', $data, $booking, $request );
$data = apply_filters( 'wpss_rest_prepare_rental', $data, $rental, $request );
$data = apply_filters( 'wpss_rest_prepare_download_grant', $data, $grant, $request );
$data = apply_filters( 'wpss_rest_prepare_service', $data, $service, $request );
```

Naming convention: `wpss_rest_prepare_{resource}`. The `extend_service_response()` and `extend_order_response()` methods on `ServiceTypeInterface` are the **type's contribution** to these filters — each type hooks into its own filter to add type-specific fields.

**5. Query args filters:**

Every listing endpoint accepts a filterable query args array so sites can pre-filter:

```php
$args = apply_filters( 'wpss_orders_query_args', $args, $request );
$args = apply_filters( 'wpss_bookings_query_args', $args, $request );
$args = apply_filters( 'wpss_rentals_query_args', $args, $request );
$args = apply_filters( 'wpss_services_query_args', $args, $request );
```

Naming convention: `wpss_{resource}_query_args`. Site-level filters can inject default constraints (e.g., only show active services, only return orders from the last 90 days by default).

**6. Pagination — `has_more` calculation rule:**

This is the single most common pagination bug across plugins. The rule:

```php
// ✅ CORRECT
$total    = $wpdb->get_var( "SELECT COUNT(*) FROM ..." );
$has_more = ( $offset + count( $rows ) ) < $total;

// ❌ WRONG — breaks when last page is exactly full
$has_more = count( $rows ) === $limit;
```

Every listing endpoint MUST run a COUNT query for accurate `has_more`. No exceptions. The base `RestController::paginated_response()` helper enforces this — new controllers extend it, never hand-roll pagination.

**7. Response envelope standard:**

List endpoints return:
```json
{
    "items":    [...],
    "total":    142,
    "page":     1,
    "per_page": 20,
    "has_more": true
}
```

Plus headers:
- `X-WP-Total: 142`
- `X-WP-TotalPages: 8`

Both are set by `paginated_response()`. Never return bare arrays for listing endpoints.

**8. Error shape:**

All `WP_Error` responses carry structured data:
```php
return new WP_Error(
    'wpss_booking_slot_taken',
    __( 'That slot is no longer available.', 'wp-sell-services' ),
    array(
        'status' => 409,
        'slot_id' => $slot_id,
        'alternative_slots' => $alternatives,
    )
);
```

The `code` field is namespaced (`wpss_`), the `message` is translatable, and `data` carries machine-readable context for client UX (alternative slots, retry hints, etc.).

**9. Return types and `WP_Error`:**

Methods that can fail return `true|WP_Error` or `array|WP_Error`, **never `void` in a union** (PHP disallows it). Callers check `is_wp_error()` BEFORE falsy checks:

```php
// ✅ CORRECT
$result = $service->create_something( $args );
if ( is_wp_error( $result ) ) {
    return $result;
}

// ❌ WRONG — misses the WP_Error case
if ( ! $result ) {
    return new WP_Error( 'failed', 'Something went wrong' );
}
```

When adding `WP_Error` to an existing method's return type, update **all callers in the same commit**. Parent class widened BEFORE child overrides (PHP covariance rule).

#### Non-negotiable admin rules (from wp-plugin-development)

These 10 rules are NOT optional. Every admin surface must comply:

**Rule 1 — Submenus = data, Settings = config.**
- Bookings submenu (has rows: slots, sessions, reservations) = **submenu**
- Downloads submenu (has rows: files, grants) = **submenu**
- Rentals submenu (has rows: availability periods, active rentals) = **submenu**
- Service Types settings (no rows, just toggles + defaults) = **Settings tab**

**Rule 2 — Logical menu order.**

Single `admin_menu:999` hook reorders `$submenu[$parent_slug]` to:

```
WP Sell Services (parent)
├── Overview                    — dashboard
├── Services                    — service list (content)
├── Orders                      — order list (content)
├── Bookings                    — booking slots + sessions (content, conditional)
├── Rentals                     — rental availability + stays (content, conditional)
├── Downloads                   — download files (content, conditional)
├── Buyer Requests              — reverse marketplace (content)
├── Vendors                     — vendor management (users)
├── Moderation                  — review/dispute queue (moderation)
├── Earnings                    — commission + wallet (monetization)
├── Withdrawals                 — withdrawal requests (monetization)
├── Analytics                   — reports (insights)
├── Audit Log                   — forensic trail (insights — 1.1.0)
├── Tools                       — import/export, cleanup (tools)
└── Settings                    — all config (config, always last)
```

Pro items cluster by function, not interleaved. A vendor looking for their bookings knows exactly where to find them (Content group, between Orders and Rentals).

**Rule 3 — Suppress 3rd-party admin notices on plugin pages.**

On every WP Sell Services admin page, hook `in_admin_header:1` to remove callbacks from `admin_notices` / `all_admin_notices` / `user_admin_notices` / `network_admin_notices`. Preserve only:
- Callbacks from our own plugin (class names contain `WPSellServices` or `WPSellServicesPro`)
- WordPress core closures (namespace check)

Prevents third-party plugin upsell noise on customer admin screens.

**Rule 4 — Single wrapper for tab content.**

The core settings page wraps `do_action('wpss_settings_tab_content')` in exactly ONE `.wpss-settings-body` div that provides uniform padding. Renderers emit raw `<section>` / `<table class="form-table">` markup with ZERO padding of their own. Never let individual tab renderers handle spacing — they must inherit from the wrapper.

**Rule 5 — Free AND Pro tab renderers use the same structure.**

When Pro adds tabs via the `wpss_settings_tabs` filter, render methods emit the same `.wpss-settings-block` pattern as Free tabs. No bare `<h3>` + `<table>` — always use the block container.

**Rule 6 — Enqueue admin CSS via prefix match.**

```php
add_action( 'admin_enqueue_scripts', function ( string $hook ): void {
    $screen = get_current_screen();
    if ( ! $screen ) return;
    if ( strpos( $screen->id, 'wpss_page_' ) !== 0
         && $screen->id !== 'toplevel_page_wpss' ) {
        return;
    }
    wp_enqueue_style( 'wpss-admin', ... );
    wp_enqueue_script( 'wpss-admin', ... );
} );
```

Never maintain a hardcoded allowlist of screen IDs — new submenus auto-load styles via the prefix match.

**Rule 7 — No raw `-1` in number inputs.**

The `download_limit`, `expires_after_days`, `max_services`, `max_featured` fields use `0 = unlimited` internally. The admin UI renders an "Unlimited" checkbox that disables the number field when checked. On save, the field value is coerced to 0 if the checkbox is enabled, preventing confused admins from typing `-1` or `0` and wondering what it means.

```
[x] Unlimited downloads
[   ] Set limit: [____]   ← greyed out when checkbox is checked
```

**Rule 8 — Role labels without slugs.**

Use `translate_user_role($role_label)` when displaying role names. No `<code>{slug}</code>` dev noise in the UI. The vendor sees "Vendor" not "Vendor (wpss_vendor)".

**Rule 9 — Docs link button on section head.**

Every settings section header uses a flex row with `margin-inline-start:auto` for the "Learn more" link, styled as a compact button (border + padding + bg). Not an inline anchor. Consistent across all sections:

```
┌─────────────────────────────────────────────┐
│ Booking Settings          [📖 Learn more]   │
│ Default reminders, slot reservation TTL...  │
└─────────────────────────────────────────────┘
```

**Rule 10 — NEVER emit browser `alert()` or `confirm()`.**

All user feedback uses a global toast utility:

```js
wpss.toast.success( 'Booking rescheduled.' );
wpss.toast.error( 'Could not save. Please try again.' );
wpss.toast.confirm( 'Delete this file?', {
    onConfirm: () => { ... },
    onCancel:  () => { ... },
} );
```

Loaded on both admin and frontend. Integrated with ARIA live regions for screen reader accessibility. The `window.confirm()` / `window.alert()` / `window.prompt()` are banned in plugin code — flagged by a grep check in CI.

### Layer 15 — Action Scheduler events + housekeeping cron

**Architecture shift from polling cron to event-driven scheduling (Q14/Q15).**

#### Action Scheduler library

- Bundled in Free plugin via Composer: `"woocommerce/action-scheduler": "^3.7"`
- Loaded via the standard multi-plugin registration pattern so it deduplicates with WooCommerce and other AS-bundling plugins
- Creates 4 tables on first activation (`wp_actionscheduler_*`); already present on sites with WooCommerce

#### Event-driven actions (per booking/rental/download/support thread)

**On booking creation, enqueue:**
- `wpss_pro_booking_reminder_24h` at `scheduled_start_utc - 24h`
- `wpss_pro_booking_reminder_1h` at `scheduled_start_utc - 1h`
- `wpss_pro_booking_autostart` at `scheduled_start_utc`
- `wpss_pro_booking_autocomplete` at `scheduled_end_utc + grace_period`

**On booking reschedule:** unschedule the 4 old actions, re-enqueue at new times with incremented ics_sequence.

**On booking cancellation:** unschedule the 4 actions.

**On rental creation, enqueue:**
- `wpss_pro_rental_checkin_reminder` at `check_in_datetime - 24h`
- `wpss_pro_rental_checkin_arrival` at `check_in_datetime`
- `wpss_pro_rental_checkout_reminder` at `check_out_datetime - 24h`
- `wpss_pro_rental_checkout_arrival` at `check_out_datetime`

**On download purchase, enqueue (when grants have expiry):**
- `wpss_pro_download_expiry_warning` at `grant.expires_at - 7d`
- `wpss_pro_download_expiry` at `grant.expires_at`

**On reservation creation (booking or rental), enqueue:**
- `wpss_pro_{type}_reservation_release` at `reservation.expires_at`

**On support thread creation (downloads), enqueue:**

### Layer 22 — Backwards compatibility

**Hard guarantees:**

- Every existing `wpss_service` post gets `_wpss_service_type = 'standard'` on migration
- Every existing `wpss_orders` row gets `service_type = 'standard'` on migration
- Every existing `wpss_conversations` row gets `purpose = 'delivery'` on migration
- `StandardServiceType` wraps current behavior verbatim — zero semantic drift
- REST responses add the new `service_type` field but nothing else changes
- All 44 existing CLI test assertions must pass after the 1.2.0 refactor
- All existing hooks, emails, crons, templates continue working

#### Migration idempotency (wp-plugin-development compliance)

Every schema migration MUST be:

1. **Version-gated** — guarded by `SchemaManager::DB_VERSION` check so it only runs on upgrade, not on every request
2. **Idempotent** — safe to re-run multiple times with no side effects
3. **Defensive** — checks `INFORMATION_SCHEMA` before running `ALTER TABLE` to avoid errors on already-migrated sites

**Standard pattern for column additions:**

```php
private function add_service_type_column(): void {
    global $wpdb;
    $table = $wpdb->prefix . 'wpss_orders';

    // Check if column already exists via INFORMATION_SCHEMA
    $exists = $wpdb->get_var( $wpdb->prepare(
        "SELECT COUNT(*) FROM INFORMATION_SCHEMA.COLUMNS
         WHERE table_schema = %s
           AND table_name   = %s
           AND column_name  = %s",
        DB_NAME,
        $table,
        'service_type'
    ) );

    if ( $exists ) {
        return; // Already migrated, nothing to do
    }

    // phpcs:ignore WordPress.DB.DirectDatabaseQuery.SchemaChange
    $wpdb->query( "ALTER TABLE {$table}
        ADD COLUMN service_type VARCHAR(32) NOT NULL DEFAULT 'standard'
        AFTER platform" );
}
```

**Standard pattern for index additions:**

```php
private function add_service_type_index(): void {
    global $wpdb;
    $table = $wpdb->prefix . 'wpss_orders';

    $exists = $wpdb->get_var( $wpdb->prepare(
        "SELECT COUNT(*) FROM INFORMATION_SCHEMA.STATISTICS
         WHERE table_schema = %s
           AND table_name   = %s
           AND index_name   = %s",
        DB_NAME,
        $table,
        'idx_service_type'
    ) );

    if ( $exists ) {
        return;
    }

    // phpcs:ignore WordPress.DB.DirectDatabaseQuery.SchemaChange
    $wpdb->query( "ALTER TABLE {$table}
        ADD INDEX idx_service_type (service_type, status, created_at)" );
}
```

**Standard pattern for data backfill:**

```php
private function backfill_service_type(): void {
    global $wpdb;
    $table = $wpdb->prefix . 'wpss_orders';

    // Batched update to avoid locking large tables
    do {
        $updated = $wpdb->query(
            "UPDATE {$table}
             SET service_type = 'standard'
             WHERE service_type = ''
             LIMIT 1000"
        );
    } while ( $updated > 0 );
}
```

Every migration routine logs its result via `AuditLogService::log( 'schema.migration', ... )` so operators can see what ran and when.

### Layer 24 — Per-type reporting capability

The `service_type` column + compound index unlock first-class per-type reporting without any new tables or aggregation pipelines. Every report below is a single indexed query on `wp_wpss_orders`.

#### Vendor reports

**Total earnings across all types:**
```sql
SELECT SUM(vendor_earnings) AS total_earnings
FROM wp_wpss_orders
WHERE vendor_id = %d AND status = 'completed';
```

**Earnings broken down by type:**
```sql
SELECT
    service_type,
    COUNT(*) AS order_count,
    SUM(vendor_earnings) AS net_earnings,
    SUM(platform_fee) AS platform_fees_paid,
    SUM(total) AS gross_revenue
FROM wp_wpss_orders
WHERE vendor_id = %d AND status = 'completed'
GROUP BY service_type;
```

Returns one row per type the vendor has actively sold. Powers the per-type earnings pie chart in the vendor dashboard.

**Average order value by type:**
```sql
SELECT
    service_type,
    AVG(total) AS avg_order_value,
    COUNT(*) AS orders
FROM wp_wpss_orders
WHERE vendor_id = %d AND status = 'completed'
GROUP BY service_type;
```

Tells a vendor which type has the highest revenue per transaction — useful for mix optimization.

#### Admin / platform reports

**Platform revenue by type, current month:**
```sql
SELECT
    service_type,
    COUNT(*) AS orders,
    SUM(platform_fee) AS platform_revenue,
    SUM(total) AS gross_marketplace_volume
FROM wp_wpss_orders
WHERE status = 'completed'
  AND created_at >= DATE_FORMAT(NOW(), '%Y-%m-01')
GROUP BY service_type
ORDER BY platform_revenue DESC;
```

**Growth trajectory by type (month-over-month):**
```sql
SELECT
    service_type,
    DATE_FORMAT(created_at, '%Y-%m') AS month,
    COUNT(*) AS orders,
    SUM(total) AS gross_volume,
    SUM(platform_fee) AS platform_revenue
FROM wp_wpss_orders
WHERE status = 'completed'
  AND created_at >= DATE_SUB(NOW(), INTERVAL 12 MONTH)
GROUP BY service_type, month
ORDER BY service_type, month;
```

Powers the "type mix over time" chart in the admin dashboard. Shows which types are scaling and which are flat.

**Vendor specialization analysis:**
```sql
SELECT
    vendor_id,
    service_type,
    COUNT(*) AS orders,
    SUM(vendor_earnings) AS earnings
FROM wp_wpss_orders
WHERE status = 'completed'
  AND created_at >= DATE_SUB(NOW(), INTERVAL 90 DAY)
GROUP BY vendor_id, service_type
HAVING earnings > 0;
```

Reveals which vendors specialize in one type vs. diversify across many. Useful for targeted vendor outreach and platform growth strategy.

#### Analytics widgets powered by this

The Q20 analytics widgets (2 per type at release, comprehensive in 1.5.0+) all sit on top of these queries. Zero new infrastructure:

- Earnings-by-type pie chart → one `GROUP BY service_type` query
- Monthly revenue stacked bars by type → one `GROUP BY service_type, month` query
- Type mix percentage over time → derived from the growth query
- Top-performing type this quarter → one query with `ORDER BY` + `LIMIT 1`

#### Strategic value

Per-type reporting is a significant marketing feature in its own right:

- **For vendors:** "See exactly how each of your revenue streams performs" — a selling point for multi-type vendors
- **For site operators:** "Break down your marketplace by vertical" — essential for platform operators running general-purpose marketplaces
- **For competitive positioning:** Stripe Connect and Shopify charge extra for this level of reporting granularity; WP Sell Services Pro gets it free from the schema design

This capability is unlocked by a single `ALTER TABLE ADD COLUMN` in 1.2.0 and an index. No new tables, no aggregation jobs, no cache warming.

### Layer 25 — Performance scaling thresholds

This section codifies where the design scales comfortably, where it hits WordPress's known limits, and what the escape hatches are. Useful for future-us and for sales conversations with customers asking about scale.

#### Comfortable operating range

The design is optimized for these scales without any special optimization work:

| Metric | Comfortable up to |
|---|---|
| Total services (`wpss_service` posts) | 20,000 – 50,000 |
| Total orders (`wp_wpss_orders`) | 1,000,000+ |
| Active vendors | 10,000+ |
| Audit log rows (`wp_wpss_audit_log`) | 10,000,000+ |
| Booking slots per vendor | 10,000+ |
| Download grants per site | 5,000,000+ |
| Concurrent cart reservations | Thousands (bounded by reservation TTL) |
| Daily order volume | Tens of thousands |

These are not hard limits — they're the range where standard indexed queries return within ~50ms on default MariaDB settings with no additional caching.

#### Indexed hot paths

Every query in the following list uses a primary key or compound index and returns in milliseconds regardless of table size:

| Query | Index used |
|---|---|
| Fetch a single service page | `wp_posts` primary key |
| Fetch a service's packages | `wp_postmeta` standard (post_id + meta_key) |
| "My orders" for a vendor/buyer | `wp_wpss_orders` (vendor_id) / (customer_id) |
| Orders by status | `wp_wpss_orders` (status) |
| Orders by type | `wp_wpss_orders` (service_type, status, created_at) |
| Bookings for a vendor next week | `wp_wpss_pro_bookings` (vendor_id, scheduled_start_utc) |
| Is a slot reserved? | `wp_wpss_pro_booking_reservations` UNIQUE(service_id, slot_start, slot_end) |
| Rental availability for a date range | `wp_wpss_pro_rental_availability` (service_id, date_start, date_end) |
| Download token lookup | `wp_wpss_pro_download_grants` UNIQUE(access_token) |
| Audit events for an order | `wp_wpss_audit_log` (object_type, object_id, created_at) |
| Audit events by actor | `wp_wpss_audit_log` (actor_id, created_at) |
| Commission report by type | `wp_wpss_orders` (service_type, status, created_at) |

#### Known WordPress scale limits (not introduced by this plan)

These hit around 5K-10K services and are the same limits every WordPress marketplace faces:

| Operation | Problem | Standard solution |
|---|---|---|
| Full-text search across service titles/descriptions | Default `WP_Query` uses `LIKE '%keyword%'` — can't index | Integrate ElasticPress / Relevanssi / Algolia / SearchWP via existing `wpss_search_query` filter |
| Filter services by price range | Would scan `_wpss_packages` postmeta JSON | Maintain denormalized lookup table (WooCommerce pattern) |
| Live faceted search (type + price + rating + category) | Multiple meta joins become slow | Same — lookup table or search index |
| `ORDER BY` on postmeta values | Postmeta is text-value, sort is slow | Promote hot meta to dedicated columns |

**None of these are triggered in the typical buyer/vendor flow.** They only matter when building marketplace-wide discovery features at scale.

#### Escape hatch: service lookup table (deferred until needed)

If and when the scale justifies it (typically around 10K+ services with live faceted search), we add a denormalized lookup table:

```sql
CREATE TABLE wp_wpss_service_lookup (
    service_id    BIGINT UNSIGNED PRIMARY KEY,
    service_type  VARCHAR(32) NOT NULL,
    vendor_id     BIGINT UNSIGNED NOT NULL,
    min_price     DECIMAL(10,2) NOT NULL,
    max_price     DECIMAL(10,2) NOT NULL,
    rating_avg    DECIMAL(3,2) DEFAULT 0,
    rating_count  INT DEFAULT 0,
    order_count   INT DEFAULT 0,
    is_featured   TINYINT(1) DEFAULT 0,
    last_updated  DATETIME,
    KEY idx_type_price (service_type, min_price, max_price),
    KEY idx_vendor (vendor_id),
    KEY idx_rating (rating_avg, rating_count),
    KEY idx_featured (is_featured, service_type)
) ENGINE=InnoDB;
```

Rebuilt on `save_post_wpss_service` and incrementally updated when reviews/orders change. Single query powers every faceted search query. Matches WooCommerce's `wc_product_meta_lookup` pattern.

**This table is NOT in 1.2.0 scope.** It's documented as the known escape hatch for when profiler data justifies it. ~2-day additive change — no refactor required because no existing code depends on it.

#### Escape hatch: audit log partitioning (deferred)

Audit log grows linearly with marketplace activity. At 100M+ rows the composite index scans slow down. Standard solutions when needed:

1. **Range partitioning by `created_at`** — MySQL handles one month at a time
2. **Archive table** — move rows older than 1 year to `wp_wpss_audit_log_archive`, query both via UNION when forensics needs old data
3. **Move to dedicated log store** — ship audit events to a separate system (ELK, Loki, S3) at write time

None of these are needed until we're running the largest marketplace on the plugin.

#### What we're NOT vulnerable to

**WooCommerce's HPOS migration problem:** WC moved orders from `wp_posts` to a custom table to solve scale issues. **We were never there in the first place** — `wp_wpss_orders` is already a dedicated custom table with typed columns. The single biggest WordPress marketplace scale refactor doesn't apply to us.

**Postmeta autoload bloat:** Our hot data (orders, bookings, rentals, grants) lives in custom tables, not postmeta. Postmeta is only used for service-listing data (titles, packages, settings) which is fetched on-demand per service page. No eager autoloading of large blobs.

**Option table bloat:** We don't store per-order, per-booking, or per-user data in `wp_options`. Only site-level settings go there (~50 options max).

#### Real-cron requirement for bookings and rentals

The one operational requirement the design imposes: **sites running booking or rental services need real cron configured**, not wp-cron. Action Scheduler still works on wp-cron but fires events on page loads, which is unreliable for time-sensitive reminders and state transitions.

The plan includes an admin warning that detects wp-cron and prompts the operator to switch to real cron when booking/rental services first exist on the site. Without booking/rental features, wp-cron is fine for everything else (audit retention, download expiry, cleanup sweeps).

### Layer 26 — UX principles and onboarding

This section exists because the technical plan optimizes for **capability** (what the product CAN do) while the site-owner experience is about **visibility** (what they SEE on a given day). Without explicit UX safeguards, a site owner installing Pro 1.4.0 encounters a Service Types settings tab with 4 subsections, ~40 new settings, new menu items, 6 new Gutenberg blocks, 18 new email templates, new dashboard sections, and type pickers at service creation — all on day one. That's overwhelming for a site owner who just wanted to sell services.

This layer codifies the rules that prevent the overwhelm while preserving the full power for those who want it.

#### The two-layer visibility model

Separate **capability** (backend permission, what a user CAN do) from **enablement** (what's visible in the UI).

**Layer A — Capabilities (Q13 decision):**
- Auto-grant `wpss_sell_downloads`, `wpss_sell_bookings`, `wpss_sell_rentals` to the `wpss_vendor` role on Pro activation
- Vendors technically have the permission to create any type their caps allow
- Pure backend concern, invisible to UI

**Layer B — Site-level type enablement (NEW):**
- New site setting: `wpss_enabled_service_types` — array of enabled type IDs
- **Default on fresh Pro activation:** `['standard']` only
- **Default on upgrade from 1.1.0:** `['standard']` only — existing customers see no change
- Admin explicitly enables each additional type in Settings → Service Types → Enabled types
- **Disabled types are completely invisible in the UI:**
  - No subsection in the Service Types settings tab
  - No options in the service creation type picker
  - No admin menu items for that type
  - No vendor dashboard sections
  - No blocks in the Gutenberg inserter
  - No audit events in the audit filter dropdown
  - No email templates in the editor
  - No analytics widgets on the dashboard
  - No filter chips on order/service lists

A site with only `standard` enabled is indistinguishable from today's Pro plugin experience. The other types exist in the codebase but are completely hidden.

Enabling a type is a single checkbox in Settings. Disabling a type hides its UI but does NOT delete data — existing orders of that type continue to work, they just don't show up in the list of actively managed types.

#### Per-screen visibility rules

These rules apply everywhere and are non-negotiable for the release:

**Service creation wizard:**
- If exactly 1 type enabled → skip the type picker entirely, go straight to that type's wizard
- If 2+ types enabled → show the type picker as step 1 with plain-language descriptions and real examples
- Type descriptions in the picker use examples ("Lightroom presets", "yoga sessions", "photo studio rental") not jargon ("DownloadServiceType", "time-slot based")

**Admin service list:**
- "Type" column → only shown if 2+ types are enabled
- "Type" filter dropdown → same
- Type icon badges appear on rows regardless, so admins can visually distinguish even when the column is hidden

**Admin orders list:**
- Same rules as service list

**Vendor dashboard — service list:**
- If vendor has services in exactly 1 type → no tabs, no filter chips, single-type list view
- If vendor has services in 2+ types → filter chips appear (Q19 decision)
- "Create Service" button goes directly to the single type if only one is available

**Vendor dashboard — type-specific views:**
- Calendar view (bookings): only shown if the vendor has created a booking service
- Downloads manager (downloads): only shown if the vendor has created a download service
- Rental availability manager (rentals): only shown if the vendor has created a rental service
- Sections don't appear in navigation until the vendor creates their first service of that type

**Customer dashboard:**
- "My Downloads" section → only shown if the customer has downloaded anything
- "My Bookings" section → only shown if the customer has an active or past booking
- "My Rentals" section → only shown if the customer has an active or past rental
- Empty sections are hidden entirely, not shown as "no items yet"

**Admin Settings → Service Types tab:**
- "Enabled types" is the first section at the top
- Per-type subsections only appear for enabled types
- Each subsection has a "Quick setup" mode (5 essential settings) and "Advanced" mode (everything)
- Default to Quick mode, remember the admin's per-subsection preference
- "Reset to defaults" button per subsection

**Admin Settings → Emails tab:**
- Only shows email templates for enabled types
- "Standard service emails" section always visible
- Download / Booking / Rental email sections appear only when respective type is enabled

**Gutenberg block inserter:**
- All blocks registered via `register_block_type` regardless of state (needed for existing post content to render)
- Block inserter visibility filtered via the `allowed_block_types_all` filter — blocks for disabled types are hidden from the inserter but still render if they're in existing content
- Prevents inserter clutter without breaking any existing pages that reference the blocks

**Analytics dashboard:**
- Widgets registered via existing `wpss_analytics_widgets` filter only include widgets for enabled types
- Core cross-type widgets (total revenue, order count) always visible

**Audit log viewer:**
- Event type filter dropdown includes only events from enabled types
- Disabled types' past events remain queryable via direct API call but don't clutter the UI filter

#### Upgrade experience for existing customers

**Critical rule: upgrading from 1.1.0 to 1.2.0 must not feel jarring for existing Pro customers.**

For a customer running WP Sell Services Pro today purely for services:

1. They upgrade via WordPress plugin update
2. Schema migration runs quietly: every existing service gets `_wpss_service_type = 'standard'`, every existing order gets `service_type = 'standard'`, every existing conversation gets `purpose = 'delivery'`
3. `wpss_enabled_service_types` option is set to `['standard']`
4. First admin page load after upgrade: shows a **gentle non-blocking dismissible admin notice**:

   > ✨ **WP Sell Services Pro 1.2.0 is here.**
   > Your existing services work exactly as before. This release adds support for selling digital downloads alongside your services.
   >
   > [Learn about what's new] [Enable digital downloads] [Dismiss]

5. If they dismiss or ignore the notice, **nothing changes in their experience**:
   - No type picker at service creation
   - No new settings tabs
   - No new dashboard sections
   - No new menu items
   - No new emails in the editor
   - Their existing workflow is untouched
6. Only when they click "Enable digital downloads" in the notice (or visit Settings → Service Types manually) does the new UI appear.

Same pattern for Pro 1.3.0 (bookings) and Pro 1.4.0 (rentals). Each release ships with a dismissible notice highlighting the new capability, but the default state is "nothing visible until enabled."

**The upgrade is silent unless the admin actively opts in.** This is non-negotiable.

#### The complexity budget

Every new UI element on a screen has a cost. The rule: **if adding something doesn't make the common case clearly better, it gets hidden behind a toggle, moved to Advanced mode, or deferred until the admin actively asks for it.**

Practical applications:

- **Default settings are set once and forgotten.** No admin should have to configure anything before using a newly-enabled type. Sensible defaults are baked in.
- **Advanced mode is opt-in per section.** Toggling Advanced is remembered per admin per section. No global flag.
- **Feature discovery is timed.** New features are surfaced in the UI only when the admin is in a context where they'd matter (e.g., booking analytics widget only after the admin's first completed booking order).
- **Settings with no clear default belong in Advanced.** Cancellation policies, meeting URL strategies, retention periods — all Advanced by default because the sensible default works for most.
- **Error messages are specific.** "Booking slot collision" not "Database error". The admin should be able to fix any error based only on the message, without reading docs.

#### Feature discovery (progressive, not all-at-once)

Rather than a firehose of "here's everything the plugin can do," the admin experience drips features as the admin needs them:

| Trigger | Feature revealed |
|---|---|
| First-run wizard | Type selection + immediate enablement |
| Admin visits Services page with zero services | "Create your first service" CTA with type-specific examples |
| Admin creates first service of type X | Type X's advanced settings become visible, demo content offered |
| Admin receives first order of type X | Order workflow email tutorials unlock |
| Admin's first week | Tips drip email sequence based on their type focus |
| Admin has 10+ orders | Analytics widgets unlock + dashboard preview |
| Admin has multiple types with data | Filter chips appear in dashboards |
| Admin hits a scale threshold (100+ services, 1000+ orders) | "Consider enabling..." suggestion for performance optimizations |

**Features don't appear until they're useful.** This reduces cognitive load and creates natural discovery moments.

#### Documentation organized by persona, not by type

Docs structure should match how customers think, not how the codebase is organized:

**Persona-first navigation:**

- Starting a **freelance services marketplace** → guide
- Starting a **digital products marketplace** → guide
- Starting a **coaching & consulting marketplace** → guide
- Starting a **space rental marketplace** → guide

The type-first reference docs still exist for developers but aren't the primary entry point for site owners.

#### Metrics to catch UX regressions

These metrics are tracked starting Pro 1.2.0 launch and used to detect if we over-complicated something:

| Metric | Target | What it means if it fails |
|---|---|---|
| Support tickets asking "how do I disable X" in 30 days | **0** | We exposed features the admin didn't opt into |
| Support tickets asking "what does X type do" | **< 10/week** | Feature descriptions aren't clear enough |

#### What to build differently if we ignored this layer

Without the UX safeguards above, the likely shipped experience is:

- ❌ Settings page with every option always visible
- ❌ Type picker appears at service creation regardless of what's enabled
- ❌ Vendor dashboard always shows 4 tabs even when they only sell one type
- ❌ Customer dashboard shows "My Downloads" / "My Bookings" / "My Rentals" to users who have never bought any of those
- ❌ Gutenberg inserter full of blocks for features the site doesn't use
- ❌ Email editor with 29 emails to customize (existing 11 + 18 new)

Every one of those is a real UX failure pattern seen in WordPress plugins that expanded scope without UX discipline (Dokan, WooCommerce addons, LearnDash's addon ecosystem, etc.).

**With the UX safeguards above, the same feature set feels focused, guided, and approachable** because at every screen the admin only sees what's relevant to them right now.

### Layer 27 — wp-plugin-development compliance reference

This layer consolidates the remaining rules from the `wp-plugin-development` skill that don't fit cleanly into a previous layer. Every rule here is mandatory and audited during code review.

#### Hook naming conventions

Every write operation (create / update / delete) on every new entity gets two hooks: a `before_` filter that can abort the operation, and an `after_` action that signals completion.

**Naming pattern:**
- `wpss_before_{operation}_{entity}` — filter, return `WP_Error` to abort
- `wpss_after_{operation}_{entity}` — action, fires after success

**Required hooks for 1.2.0 scope:**

Orders (Free — already partially exists, extended in 1.2.0):
- `wpss_before_create_order` (filter), `wpss_after_create_order` (action)
- `wpss_before_update_order` (filter), `wpss_after_update_order` (action)
- `wpss_before_delete_order` (filter), `wpss_after_delete_order` (action)
- `wpss_before_cancel_order` (filter), `wpss_after_cancel_order` (action)

Services (Free — extended in 1.2.0):
- `wpss_before_create_service` (filter), `wpss_after_create_service` (action)
- `wpss_before_update_service` (filter), `wpss_after_update_service` (action)
- `wpss_before_delete_service` (filter), `wpss_after_delete_service` (action)
- `wpss_before_publish_service` (filter — runs when type becomes immutable per Q3)

Bookings (Pro 1.3.0):
- `wpss_before_create_booking`, `wpss_after_create_booking`
- `wpss_before_reschedule_booking`, `wpss_after_reschedule_booking`
- `wpss_before_cancel_booking`, `wpss_after_cancel_booking`
- `wpss_before_reserve_slot`, `wpss_after_reserve_slot`
- `wpss_before_release_slot`, `wpss_after_release_slot`

Rentals (Pro 1.4.0):

**`before_` filter contract:**

```php
/**
 * Filter whether an order can be created.
 *
 * @param true|\WP_Error $allow      True to allow, WP_Error to abort with reason.
 * @param array          $order_data Order data being created.
 * @param int            $actor_id   Current user ID.
 */
$allow = apply_filters( 'wpss_before_create_order', true, $order_data, $actor_id );

if ( is_wp_error( $allow ) ) {
    return $allow;
}
if ( false === $allow ) {
    return new WP_Error( 'wpss_order_creation_blocked', __( 'Order creation was blocked.', 'wp-sell-services' ) );
}
```

**`after_` action contract:**

```php
/**
 * Fires after an order is created.
 *
 * @param int   $order_id   The new order ID.
 * @param array $order_data Data used to create the order.
 * @param int   $actor_id   Current user ID.
 */
do_action( 'wpss_after_create_order', $order_id, $order_data, $actor_id );
```

Every hook signature documented in `docs/ARCHITECTURE.md` so developers know what arguments to expect. Inconsistent signatures across call sites are a bug that cost us two days in 1.1.0 (the `log_status_change` arg-order bug) — this layer exists to prevent it.

#### URL and SEO safety

**Base slug changes need redirects:**

If any release changes a public URL pattern (e.g., renaming `/services/` to `/marketplace/`), the migration:
1. Stores the old slug in `wp_options` keyed as `wpss_legacy_slugs[]`
2. Registers a 301 redirect handler on `template_redirect` that matches old slugs and redirects to the new slug
3. The redirect handler lives indefinitely — never removed — to avoid breaking inbound links from customer marketing

**Rewrite flush is deferred to `init:99`, never during activation:**

```php
add_action( 'init', static function (): void {
    if ( get_transient( 'wpss_flush_rewrite_rules' ) ) {
        flush_rewrite_rules( false );
        delete_transient( 'wpss_flush_rewrite_rules' );
    }
}, 99 );
```

Activation hook sets the transient; `init:99` consumes it. Flushing during activation is unsafe because WordPress's rewrite API isn't fully loaded yet.

**Short URL prefixes to avoid WordPress core route collision:**

New URL patterns for 1.2.0-1.4.0 use short prefixes to reduce collision risk with WordPress core or other plugins:

| Concept | URL pattern | Rationale |
|---|---|---|
| Download file serving | `/wpss-dl/{token}` | Public route, rate-limited, audit-logged |
| Booking details | `/b/{booking_uid}` | Short for ICS links and SMS reminders |
| Rental details | `/r/{rental_uid}` | Same |
| Vendor calendar subscription | `/wpss/schedule/{vendor_slug}.ics` | Namespaced, content-type aware |

`/wpss-dl/`, `/b/`, `/r/` — all short, unlikely to collide with core WordPress routes (`/feed/`, `/page/`, `/comments/`, etc.) or common plugin routes.

#### Unbounded table pruning

Every table that grows without bound has a daily cron sweep with a configurable retention:

| Table | Cron | Default retention |
|---|---|---|
| `wp_wpss_audit_log` | `wpss_audit_log_cleanup` (already shipped 1.1.0) | 0 = keep forever |
| `wp_wpss_pro_booking_reservations` | Released via Action Scheduler event at `expires_at`; daily sweep as backup | 15 minutes TTL |
| `wp_wpss_pro_rental_reservations` | Same | 15 minutes TTL |
| `wp_wpss_pro_download_grants` (expired) | `wpss_pro_download_expiry_cleanup` | Configurable, default 0 = keep |
| `wp_wpss_messages` (support threads on closed downloads) | `wpss_pro_support_thread_autoclose` | 90 days after inactivity |

Every retention is an admin setting; operators can tune per compliance needs. No table is allowed to grow without bound.

#### Return type conventions

- Methods that can fail: `public function create_order( array $data ): int|WP_Error`
- Methods that can succeed/fail without return value: `public function delete_order( int $id ): true|WP_Error` (NEVER `void|WP_Error` — PHP forbids void in unions)
- Parent class return type widened BEFORE child overrides (PHP covariance rule)
- When adding `WP_Error` to an existing method's return type, update all callers in the same commit
- Callers check `is_wp_error()` BEFORE falsy checks — this is a common bug

### Layer 28 — Three-layer testing workflow

This layer replaces the plan's existing "CLI tests" mention with a complete testing methodology aligned with `wp-plugin-development`. Three layers of coverage are mandatory for every plan item.

#### The problem code quality tools can't catch

WPCS, PHPStan, PHPUnit, and CI pass when the **code that exists is correct**. They cannot catch:
- Planned features that were never implemented
- Missing CSS rules, spacing, or hover states
- Theme conflicts on styled anchor elements
- UI elements the plan specified but code forgot
- Broken cross-layer wiring (JS action name doesn't match PHP handler)

Unit tests pass even when half the plan is unimplemented — they only test what you wrote, not what you should have written. A plan item "Add boost button to feed cards" with passing CI but no actual button in the DOM is a real failure mode.

#### The three layers

Every plan item needs coverage at ALL THREE layers. Missing any layer creates a gap:

| Layer | What it catches | Tool |
|---|---|---|
| **Acceptance tests** (plan-derived) | Missing functionality — planned features that don't exist | PHPUnit functional tests |
| **Browser verification** (per item) | Visual gaps — wrong CSS, theme conflicts, broken layout | Playwright MCP |
| **Code quality** (CI) | Bad code — syntax, standards, type errors | WPCS + PHPStan + Lint |

Code quality alone = half-cooked delivery. All three = fully baked.

#### Plan-derived acceptance tests — write BEFORE code

Every plan item becomes a PHPUnit acceptance test written BEFORE implementation. The test should FAIL initially, then pass as code lands.

**Workflow:**

```
Plan phase:
    Plan item → Write failing acceptance test (RED)

Implementation phase:
    Write code → Acceptance test PASSES (GREEN)
                 Browser verify via Playwright
                 Fix visual gaps
                 THEN mark item done
```

**Test patterns by plan item type:**

**New UI element:**

**New REST endpoint:**
```php

    $service_id = $this->create_download_service();
    $order_id = $this->create_paid_order( $this->buyer_id, $service_id );

    wp_set_current_user( $this->buyer_id );
    $response = $this->rest_get( "/wpss/v1/orders/{$order_id}/download-files" );
```

#### Per-item browser verification (after acceptance tests pass)

After the PHPUnit acceptance tests pass for a plan item, verify visually via Playwright MCP:

```
1. Navigate to the affected page
2. Screenshot the area (full page or element)
3. Interact: hover buttons, click actions, submit forms
4. Compare against what the plan specified
5. Fix visual/CSS gaps → re-screenshot → THEN mark item done
```

**Do NOT batch all items then verify at the end. Verify per item.**

Browser verification catches what acceptance tests can't:
- Visual regressions (wrong padding, broken border, misaligned icon)
- Theme conflicts (hover color inherited from theme link color)
- Responsive breakage (layout breaks at 390px)
- Missing CSS (class exists in HTML but no styles applied)
- Missing interactivity (button exists but click doesn't do anything because JS isn't wired)

#### What to verify per change type

| Change type | Acceptance test (PHPUnit) | Browser verify (Playwright) |
|---|---|---|
| **New UI element** | Template contains expected CSS class/markup | Screenshot + hover/focus/visited states |
| **REST endpoint** | Response shape + auth + error cases | UI that consumes it works end-to-end |
| **Admin page** | Form renders, save handler works, edit prefills | Screenshot form/list + interactions |
| **Interactivity API store** | N/A (JS) | Navigate → interact → screenshot result |
| **Filter/hook** | `apply_filters()` returns expected result | Feed/page shows modified output |
| **CSS-only change** | N/A | Screenshot normal + hover + focus + visited + 390px + dark mode |
| **Email template** | Email service renders correct subject/body with test data | Render in email preview, check on mobile mail clients |
| **ICS attachment** | ICS file parses correctly, VTIMEZONE block valid | Open in calendar app, confirm event appears |
| **Action Scheduler event** | Event scheduled with correct timestamp + args | Event fires in admin Tools → Scheduled Actions |
| **Audit log integration** | Audit row written with correct event_type + context | Row appears in REST `/audit-log` response |

#### Theme conflict prevention checklist

For every `<a>` element styled as a button in our templates:
- [ ] Normal state: text color is our brand color, not theme link color
- [ ] `:hover` state: text color unchanged (not theme hover link color)
- [ ] `:visited` state: text color unchanged (not purple)
- [ ] `:focus` state: focus outline is our outline, not theme outline
- [ ] Explicit `color` declaration on the anchor, not `color: inherit`
- [ ] Tested on 2+ themes (Twenty Twenty-Four and one popular third-party)

#### The complete workflow per plan item

```
1. PLAN        → Plan item documented with verification checklist
2. TEST FIRST  → Write PHPUnit acceptance test (should FAIL — red)
3. IMPLEMENT   → Write code until acceptance test PASSES (green)
4. BROWSER     → Playwright MCP verify the item visually
5. FIX         → Fix any visual/CSS gaps found in step 4
6. DONE        → Mark item complete only after steps 3+4+5 pass
```

**If you skip step 2, you deliver half-cooked code with passing CI.** CI checks code quality, not feature completeness. The acceptance test is the feature completeness check.

#### Scope impact

Adding Layer 28 increases engineering time per feature by ~20-30%. This is not optional — it's the difference between "code ships and 30% of plan items were silently skipped" and "code ships with full feature coverage and fewer post-launch bugs."

The 20-30% overhead is more than offset by:
- Fewer post-launch bug fixes (caught in test before ship)
- Faster onboarding for new developers (tests document expected behavior)
- Safer refactoring (tests catch regressions immediately)
- Real confidence in "done" status (not "done" with missing UI)

**Every 1.2.0/1.3.0/1.4.0 release must ship with full three-layer coverage.** Releases that skip acceptance tests or per-item browser verification are not considered done.

## Risk log

| Risk | Mitigation |
|---|---|
| Schema migration on large customer sites is slow | Batched UPDATEs with row limits; runs once during `plugins_loaded` on version change |
| Existing customer hooks break from changed behavior | `StandardServiceType` preserves current semantics verbatim; 44-assertion regression guard |
| Cart mixing edge cases (refund, commission) | Each cart item independently processed through its type strategy; commission engine unchanged |
| Booking/rental reservation races | UNIQUE indexes on reservation rows + Action Scheduler release events |
| Download signed URL abuse | TTL on URLs + download count enforcement + audit every serve |
| Support thread spam | Existing `RateLimiter` applies |
| Action Scheduler loading conflicts with other plugins | Standard bundle pattern deduplicates automatically with WooCommerce and other AS-bundlers |
| Timezone / DST boundary bugs | Store UTC, render per-recipient, use IANA names + `DateTimeZone::getTransitions()` for ICS VTIMEZONE |
| Cron precision on low-traffic sites without real cron | Admin warning banner requiring real cron for booking/rental services |
| Backwards compatibility regression | Full CLI regression suite + time-machine test + manual browser pass per release |
