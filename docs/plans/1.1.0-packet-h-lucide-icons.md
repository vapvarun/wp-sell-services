## Why

Per `~/.claude/skills/wp-plugin-development/references/admin-ux-rulebook.md`,
all Wbcom plugins must use **Lucide icons only** across frontend AND
backend. No new Dashicons, no raw inline SVG for icon purposes.

Current audit on branch `1.1.0`:
- 196 Dashicons usages in PHP templates / services
- 128 raw `<svg>` blocks (some are ours from 1.1.0 work — milestone flag,
  extension clock, tip heart, lock, approve checkmark)
- Zero Lucide wiring today

Parallel goal: every icon, colour, spacing, radius, font-size must flow
through a CSS custom property so a white-label theme can retint by
setting `:root` overrides. Packet D is already consolidating the colour
tokens; Packet H extends that rule to icons.

## Rules

1. **No `dashicons-*` class** in any new or edited file. Each dashicon becomes:
   ```html
   <i data-lucide="{lucide-name}" class="wpss-icon wpss-icon--sm"></i>
   ```

2. **No inline `<svg>` for icons.** Same replacement pattern. Keep inline SVG
   only for logos / illustrations that aren't an icon (tip heart is an icon — must
   migrate; a project-complete trophy illustration may stay).

3. **Lucide library loaded once** per surface (frontend-once, admin-once):
   ```php
   wp_enqueue_script( 'lucide', WPSS_PLUGIN_URL . 'assets/js/vendor/lucide.min.js', array(), '0.460.0', true );
   wp_enqueue_script( 'wpss-icons', WPSS_PLUGIN_URL . 'assets/js/wpss-icons.js', array( 'lucide' ), WPSS_VERSION, true );
   ```

4. **Bootstrap** (`wpss-icons.js`):
   ```js
   document.addEventListener('DOMContentLoaded', function () {
       if (window.lucide) lucide.createIcons();
   });
   document.addEventListener('wpss:icons:refresh', function () {
       if (window.lucide) lucide.createIcons();
   });
   ```
   The custom `wpss:icons:refresh` event lets dynamically-inserted markup
   (modal bodies, AJAX responses) re-hydrate their icons — dispatch it
   after every successful milestone / extension / tip modal open, after
   every timeline reload.

5. **Size tokens** (add to `design-system.css`):
   ```css
   :root {
       --wpss-icon-size-sm: 14px;
       --wpss-icon-size: 18px;
       --wpss-icon-size-lg: 24px;
       --wpss-icon-stroke: 1.75;
   }
   .wpss-icon            { width: var(--wpss-icon-size); height: var(--wpss-icon-size); stroke-width: var(--wpss-icon-stroke); }
   .wpss-icon--sm        { width: var(--wpss-icon-size-sm); height: var(--wpss-icon-size-sm); }
   .wpss-icon--lg        { width: var(--wpss-icon-size-lg); height: var(--wpss-icon-size-lg); }
   ```
   All `[data-lucide]` renders use these token-driven classes. No inline
   `width="24"` attributes on new icons.

6. **Mapping reference** (our 1.1.0 icons to Lucide names):
   - Tip heart → `heart`
   - Extension clock → `clock-alert` (or `clock-3`)
   - Milestone flag → `flag` (our `M→` flag SVG) → `flag`
   - Lock (locked phase) → `lock`
   - Approve / completed check → `check-circle-2`
   - Project-complete banner → `party-popper` (or `trophy`)
   - Send / deliver → `send`
   - Decline / cancel → `x-circle`
   - View → `eye`
   - Edit → `pencil`
   - Delete → `trash-2`
   - Download CSV → `download`
   - Upload / request withdrawal → `upload`
   - Chat / message → `message-square`
   - Video intro → `video`
   - Wallet → `wallet`
   - Commission / money → `banknote`
   - Analytics → `chart-column`
   - Calendar / due date → `calendar-clock`
   - Info → `info`
   - Warning → `triangle-alert`

7. **Dashicons fallback shim.** In the bridge period, keep a single CSS
   fallback rule so any unmigrated Dashicon still renders its old glyph:
   ```css
   .dashicons.dashicons-money-alt::before { /* optional visible fallback */ }
   ```
   Or easier: leave Dashicons installed (it's WP core), migrate everything
   in one commit, ship. No shim needed.

## Acceptance criteria

1. `grep -rn "dashicons-" templates/ src/ | wc -l` returns **0** in both
   plugins after the packet.
2. `grep -rn "<svg" templates/ | wc -l` drops to the count of legitimate
   illustrations (not icons). All icon `<svg>`s replaced with
   `<i data-lucide="...">`.
3. Lucide renders on:
   - Dashboard sales page
   - Dashboard wallet page
   - Dashboard analytics page
   - Milestone timeline + propose modal
   - Milestone / extension / tip receipt views
   - Proposal form modal
   - Service detail page (seller card + intro video section)
   - Admin Settings page
   - Admin Vendors list (Contract Types column indicator icon)
4. Every icon is tinted via `currentColor` — no hardcoded fill / stroke
   attributes inside the `<i>`.
5. Every modal that renders dynamically triggers
   `document.dispatchEvent(new CustomEvent('wpss:icons:refresh'))` on
   open.
6. No visible regression: icons render at the same approximate size and
   position as the Dashicon / SVG they replaced. Screenshots of 10
   representative surfaces before + after in
   `docs/qa/screenshots/lucide-migration/`.
7. Token-ready proof: override `--wpss-icon-size` to `32px` via a
   `:root` block and confirm every icon scales uniformly. Screenshot as
   `docs/qa/screenshots/lucide-migration/theme-override-test.png`.

## Extended audit — other skill rules to sweep alongside icons

These all live in the same `wp-plugin-development` skill. Compliance is
cumulative.

### 1. Token scale alignment (`frontend-tokens.md`)

Our `design-system.css` uses `--wpss-*` tokens — that's fine (prefix
matches the plugin). But the scale itself should match the skill's
named scale rather than ad-hoc values:

- Spacing: `xs / sm / md / lg / xl / 2xl / 3xl / 4xl` — audit our
  `--wpss-space-1` … `--wpss-space-12` and add semantic aliases.
- Typography: `xs / sm / base / md / lg / xl / 2xl / 3xl` — same.
- Font family is ALWAYS `inherit` — search for `font-family:` rules and
  confirm no plugin CSS forces a family.
- Every card padding = `--wpss-space-xl`; grid gap = `--wpss-space-xl`;
  inline icon-to-text gap = `--wpss-space-sm`. Audit new components
  (milestone timeline, proposal form, ledger) for compliance.

### 2. Button + card + badge component rules (`frontend-components.md`)

- Buttons: height `32px` (compact) / `40px` (standard) / `44px` (large)
  — confirm `.wpss-btn--sm` / `.wpss-btn` / `.wpss-btn--lg` match.
- Card: standard internal padding `--wpss-space-xl`.
- Badges (our tip / extension / milestone pills): check padding
  `2px var(--wpss-space-sm)`, font `--wpss-text-xs`, weight
  `--wpss-font-semibold`, radius `--wpss-radius-pill`.

### 3. Responsive rules (`frontend-responsive.md`)

Packet C landed mobile rules but did not audit against the full skill.
H verifies:
- Mobile-first where feasible (base rules = mobile, `@media (min-width: ...)` for tablets+).
- Breakpoints match the skill's named stops (e.g. `--wpss-breakpoint-sm`
  = 640px, `-md` = 768px, `-lg` = 1024px). If we're using arbitrary
  pixel values, refactor.

### 4. Accessibility pass (`frontend-accessibility.md`)

Milestone timeline + proposal form + receipts + ledger — walk each
with a keyboard-only:
- Every interactive element is focusable in order.
- Every button has a visible `:focus-visible` outline.
- Every modal traps focus when open + releases on close.
- Every icon-only button has `aria-label`.
- Every form field has an associated `<label>`.
- Contrast ratio ≥ 4.5:1 for text on backgrounds, ≥ 3:1 for UI glyphs.

Our new surfaces (`wpss-milestone-wrap`, `wpss-btn--locked`,
`wpss-milestone-decline-btn`, etc.) need the once-over.

### 5. Security pass (`security.md`)

Re-sweep the 1.1.0 AJAX handlers + REST endpoints for:
- Nonce verification on every mutation (`check_ajax_referer` / REST
  `permission_callback`).
- Capability check alongside ownership (not just "is the right user id").
- `$wpdb->prepare()` on every SQL that interpolates user-controlled
  values. Packet F touched `VendorsPage.php` with new correlated
  subqueries — confirm they're parameterised.
- Output escaping: `esc_html` / `esc_attr` / `esc_url` / `wp_kses_post` on
  every template echo. The `milestone-view.php`, `extension-view.php`,
  and the proposal modal repeater are the highest-risk files.

### 6. Hook lifecycle (`lifecycle.md`)

Every write operation must fire `before_` (filterable, `WP_Error`
abortable) and `after_` (action) hooks. Audit:
- `MilestoneService::propose` / `submit` / `approve` / `decline` /
  `delete_unpaid` / `credit_milestone_on_payment_complete`
- `ExtensionOrderService::create_extension_request` /
  `credit_extension_on_payment_complete` / `decline`
- `TippingService::create_pending_tip_order` /
  `credit_tip_on_payment_complete`
- `ProposalService::submit` / `update`
- `BuyerRequestService::convert_to_order`

Add any missing hooks. Skill example:
```php
$allow = apply_filters( 'wpss_before_milestone_propose', true, $parent_order_id, $vendor_id, $data );
if ( is_wp_error( $allow ) ) { return $allow; }
// ... do the work ...
do_action( 'wpss_after_milestone_propose', $milestone_id, $parent_order_id, $vendor_id, $data );
```

### 7. REST contract (`wbcom-architecture.md` §3)

Packet B shipped the REST milestone endpoints. Verify against:
- Namespace `wpss/v1` — ✅ already correct.
- Pagination: `has_more = (offset + count) < total` — NOT
  `count === limit`. Audit every list endpoint.
- COUNT query alongside the SELECT for accurate `total`.
- Response shape filter: `wpss_rest_prepare_milestone`,
  `wpss_rest_prepare_extension`, `wpss_rest_prepare_proposal` —
  allow themes + extensions to mutate.
- Permission callbacks return `WP_Error` with explicit 401 /
  403 codes, never bare `false`.

### 8. Data + cron (`data-and-cron.md`)

- Every new table: `ENGINE=InnoDB` explicitly (verify via
  `SchemaManager::get_*_table`).
- Multi-step writes wrapped in `START TRANSACTION` / `COMMIT` —
  `convert_to_order` is already transactional; confirm tip +
  extension credit paths are too when they insert wallet rows
  alongside order updates.
- Abandon-cron pruning already has 48h default + contract-milestone
  carve-out. Confirm configurable retention per skill (add
  `wpss_abandon_tip_hours` / `_extension_` / `_milestone_` filters).

### 9. Admin UX patterns (`admin-ux-rulebook.md`)

- Admin Settings page uses `jt-settings-card` (or `wpss-settings-card`)
  pattern. Audit existing Settings.php — does it match?
- Empty states use the same shell: icon + title + body + CTA.
  Milestone empty state, extension empty state, sales-orders empty state
  — do they match the skill's shell markup?
- Admin Vendors page (Packet F added Contract Types column) — audit
  column sortable state indicator, screen-reader labels.

### 10. Settings API (`settings-api.md`)

New notification toggles we shipped (milestone_proposed / _paid /
_submitted / _approved / extension_proposed / _approved / _declined):
- Registered via `register_setting()` + `add_settings_field()` — ✅
  already done in `Settings.php`.
- Each field has a `sanitize_callback` — verify.
- Description text uses `__()` — verify.

### 11. Structure (`structure.md`)

- Bootstrap file loads on `plugins_loaded` — ✅ already done.
- Admin code lazy-loaded behind `is_admin()` — audit:
  `EmailService::send_milestone_*` methods get instantiated on the
  notification listener. Are they gated correctly?

### 12. Debugging / logging (`debugging.md`)

- `WP_Error` return types instead of `false` on service methods — the
  `success/message/*` array pattern our services use works but isn't
  the skill's preferred shape. Document the divergence in
  `docs/architecture/SERVICE_CONVENTIONS.md`, or refactor to
  `WP_Error` returns for v1.2.
- Error logging via `wpss_log( $message, 'error' )` (already exists) —
  verify every `ROLLBACK` path and every missing-data guard logs.
