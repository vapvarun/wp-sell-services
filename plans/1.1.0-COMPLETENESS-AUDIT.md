# 1.1.0 Completeness Audit

> **Goal:** every existing feature in 1.1.0 works 100% end-to-end for **admin, vendor, and customer**. We are NOT adding new features in this audit. Anything found missing entirely → parked for a future release.

**Status:** Empty matrix. Fill in once Shashank wraps `1.1.0-rc3` QA so we don't double-spend his time.

**Output:** a single fix-list `feat/1.1.0-completeness` branch that closes every ⚠️ / ❌ cell (categorised below) before tagging 1.1.0.

---

## Categorisation rules

| Severity | Means | What we do |
|---|---|---|
| 🔴 **Broken** | Feature claims to do X, doesn't actually do X | Fix for 1.1.0 |
| 🟠 **Incomplete** | Works for one persona but missing another's view | Fix for 1.1.0 |
| 🟡 **Polish** | Works but UX is confusing / silent failure / dead-end | Fix for 1.1.0 |
| 🟢 **Missing** | Feature concept doesn't exist in code | Park in `plans/future-features/` |

**Bar for ✅ (no severity):** A brand-new user of this persona can complete the workflow end-to-end without asking for help, and gets clear feedback at every step (success, failure, empty state).

## How I'll fill it in

For each feature × persona cell:

1. **Spin up the persona** via `?autologin=<user>`
2. **Walk the happy path** end-to-end in browser
3. **Hit one edge case** per cell (network error, empty state, permission boundary, refresh mid-flow)
4. **Note any friction** in the corresponding row's "Notes" column with a severity emoji
5. **Take a screenshot** of any ⚠️ / 🔴 / 🟠 finding to support the fix later

Avoid speculative findings — if something looks weird but actually works, mark ✅. The earlier audits over-flagged speculative items (50%+ of CB/VS items were closed during implementation). This audit aims for a >85% real-friction rate.

## Admin overwhelm — separate sweep, same release

After the matrix is filled, do a second pass framed as: **"What lands on the admin's plate per cadence?"**

| Cadence | Admin should see |
|---|---|
| Daily | Disputes needing mediation. Vendor escalations. High-value withdrawal requests. |
| Weekly | Moderation queue. New vendor approvals (in approval mode). Commission rate review. |
| Monthly | Settings tuning. Top-line marketplace metrics. |
| Never (auto) | Routine workflow that should never reach admin. |

For each admin screen: is it surfacing things in the right cadence bucket? If admin's "today" view shows 50 things → that's the friction.

Output: a `Daily / Weekly / Monthly` mental model overlay on the existing admin pages, with bulk-actions added where missing. Either part of the same `feat/1.1.0-completeness` branch or split into `feat/1.1.0-admin-polish` if scope balloons.

---

## Out of scope

- Adding NEW features (parked for future)
- Refactoring code that works
- Changing money flow defaults (clearance days, commission rates)
- Pro plugin features beyond cross-checks that they still work
- Performance / scalability work
- Security audit (separate exercise)

---

## Release sequence

1. **Now** → Shashank tests rc3
2. **When Shashank reports** → categorise findings into the same severity buckets above
3. **Audit matrix filled in** → in parallel with Shashank's testing (read-only browser walk, doesn't disrupt his testing)
4. **Combined fix list** → `feat/1.1.0-completeness` branch
5. **`1.1.0-rc4`** with all fixes
6. **Final QA** → tag `v1.1.0` → ship

This is the LAST audit before tagging. Everything found that's a 🟢 missing-feature gets written into a separate `plans/future-features/from-1.1.0-audit.md` so we don't lose ideas, but they don't block the release.
