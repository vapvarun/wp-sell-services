# 1.1.0 Parallel Agent Task Plan

Breakdown of remaining 1.1.0 work into **independent packets** that can be
dispatched to separate agents in parallel. Each packet is:

- **Scoped** to one surface / subsystem so there is no file-edit overlap
  between agents.
- **Briefed** with exact files, exact acceptance criteria, and what NOT to
  touch.
- **Verifiable** — each task has a concrete way to prove it's done.

All work targets the `1.1.0` branch on `wp-sell-services` and
`wp-sell-services-pro`.

### Acceptance criteria

- Terminology consistency:
  - Settle on **"Phase"** as the noun across all vendor/buyer-facing UI (proposal form, timeline rows, emails, receipts).
  - **"Milestone"** stays as the system/feature name in section headings ("Milestones (N)"), email subjects, and admin settings.
  - Search all owned files for "milestone"/"Milestone" → every instance is either the feature name or inside a `__( 'Milestone ...', '...' )` translator context.
- Wallet naming:
  - Sales page card: "Revenue"
  - Analytics card: "Net Earnings"
  - Wallet page card: "Available Balance" + "Total Earned"
  - Every email / notification that refers to money moving: use "credited to your wallet" consistently. Remove "balance" / "earnings balance" mixing.
- Capitalisation: "Submit Delivery" / "Accept & Pay" are Title Case; body labels ("Pay phase 1 first") are sentence case. Grep for any mixed case and normalise.
- Action verbs: prefer "Decline" for buyer-refuse-unpaid; "Cancel proposal" for vendor-retract-unpaid; "Remove" never.
- Every string goes through `__( '...', 'wp-sell-services' )` or `esc_html__` — no hardcoded English in PHP, no hardcoded English in JS except via `wpssData.i18n?.key || 'fallback'`.
- Readability: Flesch reading ease ≥ 60 on the paragraphs (roughly: short words, short sentences).

### Prove-it tests

- `grep -rE "milestone|Milestone|phase|Phase" templates/order/milestone-view.php | wc -l` — diff before/after to confirm the refactor ran.
- Load a milestone-based order in the browser, walk through proposed → paid → submitted → approved → cancelled states and screenshot each; the copy narrates the story without jargon.
- Open WP's "Tools → Export" or similar translation export — every new string shows up in the POT (`wp-sell-services.pot`) file after regen.

## Packet B — REST + mobile app parity audit

**Agent:** 2
**Can run parallel with:** A, C, D, E, F, G (they don't touch REST files)
**Estimated scope:** ~3 hours

### Files this agent owns

- `src/API/MilestonesController.php`
- `src/API/TippingController.php`
- `src/API/ExtensionRequestsController.php`
- `src/API/ProposalsController.php`
- `src/API/OrdersController.php` — any endpoint that touches a sub-order
- `src/API/RestController.php` — base class, add shared helpers here if needed

### Acceptance criteria

1. **Milestone REST endpoints** match the dashboard AJAX surface:
   - `GET /wpss/v1/orders/{id}/milestones` returns the same payload shape the dashboard template uses (include `is_locked` computed field).
   - `POST /wpss/v1/orders/{id}/milestones` → propose (existing)
   - `POST /wpss/v1/milestones/{id}/submit` → submit
   - `POST /wpss/v1/milestones/{id}/approve` → approve
   - `POST /wpss/v1/milestones/{id}/decline` → decline (buyer-only, unpaid)
   - `DELETE /wpss/v1/milestones/{id}` → cancel proposal (vendor-only, unpaid)
2. **Extension REST endpoints:**
   - `POST /wpss/v1/orders/{id}/extension` → create quote
   - `POST /wpss/v1/extensions/{id}/decline`
3. **Proposal contract_type:**
   - `POST /wpss/v1/buyer-requests/{id}/proposals` accepts `contract_type` + `milestones[]`.
   - `GET .../proposals` returns decoded `milestones` field.
   - Milestone validation mirrors server-side normalisation.
4. Every response is:
   - JSON only (no HTML leakage)
   - Has explicit HTTP status: 200 ok / 201 created / 400 validation / 401 not logged in / 403 ownership fail / 404 not found / 409 state conflict (e.g. milestone locked)
5. Every write endpoint uses `permission_callback` with actual ownership check — not just `is_user_logged_in`.
6. Error bodies follow the existing plugin convention: `{ "code": "wpss_milestone_locked", "message": "...", "data": { "status": 409 } }`.

### Prove-it tests

Add `tests/rest/milestones-upwork-flow.php` (or equivalent) that uses the
WP test framework to:
- Create a buyer request.
- Submit a milestone proposal via REST.
- Accept it as the buyer via REST.
- Hit each milestone endpoint (propose, submit, approve, decline).
- Assert the lock-step 409 when trying to pay phase 2 before phase 1 clears.

### Hard constraint

REST-only work. No template or CSS changes.
