# 1.1.0 Data Flow Audit — Findings

Audit of the v1.1.0 feature set shipped 2026-04-21 — wallet single-source-of-truth, milestone / extension / tip sub-order flows, Upwork-style proposals, vendor video intro, 7 new email templates, and the alignment work across Sales / Analytics / Wallet surfaces.

Scope covered: the files and extension points named in the audit brief plus upstream hook producers (StandaloneOrderProvider::mark_as_paid, OrderWorkflowManager::handle_order_completed, CommissionService::record) needed to verify the flow end-to-end.

## Summary
- Critical: 3 (data integrity / money math broken)
- High: 4 (silent failures, broken paths)
- Medium: 4 (edge cases)
- Low: 2 (defensive improvements)

---

## Critical findings

### C1 — Tip idempotency key uses parent_order_id, not tip sub-order id
**File:** `src/Services/TippingService.php:314-321` (lookup) and `:405-407` (insert)

**What I expected:** identical shape to milestone/extension — `reference_type='order'`, `reference_id = <tip_sub_order_id>`. That makes the duplicate-webhook guard identical across the three sub-order flows and makes the ledger row point at the object that actually carries the money (the tip sub-order), not the parent it targets.

**What the code does:** the lookup AND the insert both key on `reference_id = $parent_order_id`. Once any tip for a given parent has been credited, the `existing` check returns the row for ANY subsequent tip sub-order on the same parent and the credit is skipped.

Today this is masked by `has_tipped()` rejecting a second tip creation on the same parent/customer pair, so only one tip sub-order ever exists per parent. But:

- The guard in `create_pending_tip_order` uses `customer_id`, not vendor or parent alone — if product ever allows multiple buyers (gift tips, team workspaces, etc.) the first tip blocks all later tips at the ledger layer even though `has_tipped()` would let the record be created.
- It diverges from the milestone/extension convention (`reference_id = sub_order_id`), making cross-surface reporting and the LedgerExporter link text inconsistent. The CSV shows `order#<parent_id>` for tips but `order#<sub_id>` for milestones/extensions — the reference column no longer uniquely identifies the event row.
- Admins reading wallet history can't jump from a tip row to the tip sub-order (which carries commission breakdown) because the reference_id points at the parent.

**Risk:** correctness is currently preserved only by a coincidence of another business rule. Any future relaxation of "one tip per customer per order" (a reasonable product request) would silently drop the second vendor payout.

**Suggested fix:** store `reference_id = $tip_order_id` (mirror milestone/extension). Update `has_tipped()` and `get_order_tip()` callers accordingly. Write a migration/backfill to rewrite existing `reference_id`s where `reference_type='order' AND type='tip'` point at the parent — or add a fallback check that looks up by `(vendor_id, type='tip')` joined against the tip sub-order's parent.

### C2 — Milestone-contract parent auto-completion bypasses OrderService and never fires `wpss_order_status_completed`
**File:** `src/Core/Plugin.php:777-788` (auto-complete listener), `src/Services/OrderService.php:236`, `src/Services/CommissionService.php:81-146`

**What I expected:** when the final milestone is approved and the parent request-order rolls up to completed, it should take the normal completion path so `CommissionService::record()`, `update_single_vendor_stats()`, the `wpss_order_status_completed` hook, and the email templates for "order completed" all run — same as every other path a service order reaches `completed`.

**What the code does:** `$auto_complete_parent` uses `$wpdb->update()` directly (`src/Core/Plugin.php:779`). It flips `status='completed'`, sets `completed_at`, and stops. It does NOT call `OrderService::update_status()`, so `wpss_order_status_changed` and the dynamic `wpss_order_status_completed` action never fire. That means:

- `OrderWorkflowManager::handle_order_completed()` — which runs `CommissionService::record()` and `update_single_vendor_stats()` — never runs for a milestone-contract parent. Today the parent has `total = 0` so commission is 0 and no wallet row is missing, but the vendor-profile counter updates (`update_single_vendor_stats`, `update_vendor_earnings`, seller-level tier work) are all skipped.
- `EmailService::send_order_completed()` (wired on `wpss_order_status_changed`) never fires for the parent. The buyer and vendor never receive the "project completed" email for milestone contracts even though they receive it for every other order type.
- Review creation — if reviews attach off the completion hook — would be broken for milestone contracts (worth checking downstream).
- Audit log `order.status_changed` entries are missing for the parent's final transition, so admin audit trails show the parent silently jumping to completed with no actor.

**Risk:** quiet data drift. Vendor stats (`total_orders`, `completed_orders`) don't include milestone-contract parents. The review prompt never reaches the buyer. Any future listener added to `wpss_order_status_completed` will have a blind spot for this class of order.

**Suggested fix:** replace the direct `$wpdb->update` with `( new OrderService() )->update_status( $parent_order_id, ServiceOrder::STATUS_COMPLETED, 'auto-completed after final milestone' )`. That fires the full hook chain and keeps the parent on the same lifecycle as every other order.

### C3 — `convert_to_order` fires `wpss_milestone_proposed` (and the buyer email) BEFORE the transaction commits
**File:** `src/Services/BuyerRequestService.php:691-728`, `src/Services/MilestoneService.php:246`

**What I expected:** if the bulk milestone-create transaction rolls back (e.g. milestone N+1 fails validation), the buyer must not have received emails or in-app notifications for the milestones that were rolled back. Side effects should be deferred until after `COMMIT`.

**What the code does:** `MilestoneService::propose()` fires `do_action('wpss_milestone_proposed', ...)` immediately after the row insert, before control returns to `convert_to_order`. The `wpss_milestone_proposed` listener in `Plugin.php:653-677` synchronously sends an in-app notification AND calls `EmailService::send_milestone_proposed()`, which calls `wp_mail()` — a real SMTP send. This happens for every milestone in the plan as the loop progresses. If milestone #3 fails and `convert_to_order` executes `ROLLBACK` on line 714, the rows for milestones #1 and #2 vanish, but their "new milestone ready for payment" emails and notifications are already delivered. The buyer then gets a Pay Now link pointing at a milestone row that no longer exists — `?pay_order=<N>` will 404 in `render_pay_order_checkout`.

**Risk:** only surfaces if a partial failure occurs (unlikely since validation already ran server-side), but the point of the transaction was precisely to protect against partial state. Partial state is still reachable from the outside even though the DB is clean.

Secondary risk: the synchronous `wp_mail()` loop inside a transaction holds the DB transaction open across N external SMTP sends — if the mail host is slow this lengthens lock hold time and increases the chance the HTTP request times out mid-transaction (the PHP fatal unwind would commit the transaction prematurely or leave the InnoDB connection in an ambiguous state depending on autocommit semantics).

**Suggested fix:** collect the proposed milestone IDs into an array inside `convert_to_order`, do not fire `wpss_milestone_proposed` from inside `propose()` while a transaction is open. After the outer `COMMIT`, iterate and fire the hook once per inserted milestone. Either add a `$fire_event` flag to `propose()` that `convert_to_order` sets to false, or move the `do_action` call to a queue that `convert_to_order` flushes post-commit.

## High findings

### H1 — Email rate-limit (5-min cooldown per recipient+type) silently drops legitimate milestone/extension emails
**File:** `src/Services/EmailService.php:1547-1553`

**What I expected:** a cooldown targeted at spammy flows (bulk new-message notifications, status-change pings) that NEVER silently discards a transaction-critical email — milestone paid, extension approved, tip received. A real vendor completing phase 1 and phase 2 minutes apart must receive both notifications.

**What the code does:** every call to `EmailService::send()` hits the same `wpss_email_cooldown_<md5(to_type)>` transient. If two milestones are paid 30 seconds apart (lock-step means milestones are often paid in quick succession), the vendor only receives the first "Milestone paid — start work" email. Worse, the method returns `false` with no logging, so the drop is invisible in the audit trail.

Real scenario: buyer accepts a milestone proposal with 4 phases, pays them rapid-fire (or the flow ran on the same wpss_order_paid webhook retry). Vendor gets one email, assumes only one phase paid, starts work on the wrong milestone.

**Risk:** not money — but directly UX-critical for the marketplace's core flow. Vendors miss work signals. Lock-step makes this MORE likely than in normal order flow because milestones are expected to fire in close succession.

**Suggested fix:** scope the cooldown to types where flooding is actually a risk (new message, status change), NOT to milestone/extension/tip/proposal events. Simplest: bypass the cooldown when `$type` is in an allow-list of "always deliver" types. Minimum add a `wpss_log('notice', ...)` so suppressed emails are visible to admins.

Alternative: key the cooldown on `(to, type, reference_id)` so two different milestone rows don't collide.

### H2 — `wpss_order_status_completed` hook is never fired — it depends on the dynamic `"wpss_order_status_{$new_status}"` action
**File:** `src/Services/OrderService.php:235-236`, `src/Services/OrderWorkflowManager.php:76`, `src/Core/Plugin.php:1461`

**What I expected:** at least one explicit `do_action('wpss_order_status_completed', ...)` call documented as the canonical firing point, OR clear documentation that it is a dynamically-generated name from `wpss_order_status_changed`.

**What the code does:** `OrderService::update_status()` fires `"wpss_order_status_{$new_status}"` dynamically (`:236`). That works for the happy path (explicit OrderService transitions). But two important paths bypass OrderService entirely:

1. The milestone auto-complete listener (see C2) uses raw `$wpdb->update`.
2. The OrderWorkflowManager `auto_complete_orders` cron (also raw DB writes in several places).

Any plugin extending via `wpss_order_status_completed` (Pro's `WalletManager::process_vendor_payout` relies on this hook — `InternalWalletProvider` peer class in `wp-sell-services-pro/src/Integrations/Wallets/WalletManager.php:91`) will miss those transitions.

**Risk:** Pro's wallet "payout to wallet on order complete" behavior misses milestone-contract parents (moot because total=0) and any other path that completes an order without going through OrderService. More subtle: the new vendor-stats updater won't run for the milestone-contract parent.

**Suggested fix:** either enforce that all status writes go through OrderService (add a strict-mode check in WP_DEBUG), or add explicit `do_action('wpss_order_status_completed', ...)` calls at the raw-write sites.

### H3 — Tip order's mid-flight `pending_requirements` status change fires status-change email to the buyer about a tip sub-order
**File:** `src/Integrations/Standalone/StandaloneOrderProvider.php:382-391`, `src/Services/TippingService.php:441-454`

**What I expected:** a tip / extension / milestone sub-order's payment confirmation should not trigger a generic "Your order is in progress" email to the buyer. The buyer is tipping — they don't have an order moving to a requirements phase.

**What the code does:** `mark_as_paid()` sets status to `pending_requirements` unconditionally (even for tip/extension/milestone platforms) and fires `wpss_order_status_changed` and `wpss_order_status_pending_requirements`. Downstream:

- `Plugin::define_notification_hooks` at `:456` → `notify_order_status()` will send a generic "status changed to pending_requirements" in-app notification to the buyer.
- `EmailService::handle_status_change` at priority 20 sends the matching email.

Then, a moment later at priority 20, TippingService/ExtensionOrderService/MilestoneService flip the status again (to `completed` / `in_progress`). Another `wpss_order_status_changed` fires (because they use `$wpdb->update`... actually they use raw wpdb — so status_changed DOESN'T fire the second time). Net result: buyer sees `pending_requirements` email/notification and never sees the corrective one.

**Risk:** UX bug, not money. Buyers get a "Complete your requirements" email for a tip they just sent.

**Suggested fix:** either `mark_as_paid()` branches on `platform` (for tip/extension/milestone skip the `pending_requirements` transition and let the sub-order service handle the final status), or the notify_order_status path early-returns when `platform` is one of the sub-order markers.

### H4 — `has_tipped()` wallet lookup uses `reference_id = $order_id` (parent), matching the key TippingService writes — this works today but depends on C1's non-standard key
**File:** `src/Services/TippingService.php:528-542`

**What I expected:** consistency with the rest of the sub-order flows — check wallet via the tip sub-order reference, not the parent.

**What the code does:** the check is aligned with `credit_tip_on_payment_complete`'s non-standard key (see C1). If C1 is fixed to use the tip sub-order id as `reference_id`, this method must be updated too — otherwise the check goes stale and a buyer can double-tip the same parent.

**Risk:** fragile coupling — fixing C1 without touching this method reopens the double-credit path.

**Suggested fix:** address alongside C1 — keep the lookup based on the tip sub-order row (which carries customer_id), not the wallet transaction row keyed on the parent.

## Medium findings

### M1 — `EarningsController::get_summary` "total_earned" sums ALL vendor_earnings across every platform including tip/extension/milestone, then diffs withdrawals to produce `available_balance`
**File:** `src/API/EarningsController.php:198-232`

**What I expected:** `available_balance` to match `InternalWalletProvider::get_balance()` byte-for-byte, since the audit brief calls out Sales/Analytics/Wallet NET alignment as a hard requirement.

**What the code does:** `total_earned = SUM(vendor_earnings WHERE status='completed')`. The ledger includes rows the orders table doesn't (admin credits via `InternalWalletProvider::credit` write a ledger row but do NOT touch `wpss_orders.vendor_earnings`). The ledger also excludes rows the orders table includes (a milestone row with `vendor_earnings` populated but wallet transaction insert failing midway — unlikely but not impossible).

**Risk:** after any admin manual credit / debit / dispute refund, the "Earnings" dashboard card and the "Wallet Balance" card will disagree. Vendors will report this as a bug and support will have to reconcile.

**Suggested fix:** make `EarningsController::get_summary` read `available_balance` from `InternalWalletProvider::get_balance()` (or the active wallet provider) and treat `total_earned` as the sum over the ledger, not the orders table. Alternatively, document that Earnings shows lifetime earnings-from-services and Wallet shows the ledger and they may differ.

### M2 — Milestone-contract parent's payment_status set to `paid` without a corresponding wallet/payment row
**File:** `src/Services/BuyerRequestService.php:629-632`

**What I expected:** `payment_status='paid'` only when money has actually moved (for a $0 parent, either set to `not_applicable` or leave at `pending` with a note, but do not claim it is "paid" because nobody paid anything).

**What the code does:** `$parent_payment_status = $is_milestone_contract ? 'paid' : 'pending';`. This sets the sentinel value but no wallet_transaction, no gateway transaction, nothing. Admin reports that filter "paid orders" will include a `$0` milestone-contract parent even though the money is on the child milestone rows.

**Risk:** minor — reporting / filter UI shows a phantom "paid" $0 order. Financial export may count it inconsistently.

**Suggested fix:** add a new `payment_status` value like `split_by_milestones` (or use `not_applicable`) for milestone-contract parents. Filters should know the parent carries no money in that case.

### M3 — Abandon-cron `NOT LIKE '%"is_contract_milestone":true%'` depends on `wp_json_encode` output format
**File:** `src/Services/MilestoneService.php:614-629`

**What I expected:** the filter to survive whitespace / ordering changes in the JSON output. `wp_json_encode` respects the `JSON_UNESCAPED_SLASHES` flag by default in recent WP but does not pretty-print. The substring match would miss if any caller ever passes `JSON_PRETTY_PRINT`, and it matches greedily against any substring — a text description field containing the literal string `"is_contract_milestone":true` would false-positive.

**What the code does:** inspected the insert site (`src/Services/MilestoneService.php:214-222`) — `wp_json_encode` is called on an associative array with `is_contract_milestone` always last. On standard WP, this produces `"is_contract_milestone":true}` as the tail of the JSON. The LIKE pattern matches. The contract-marker flag does work in practice today.

However the match is unanchored — if a buyer-supplied `description` field ever contained the literal substring (easy to forge: name your milestone `"is_contract_milestone":true`), the abandon-cron would skip cleaning up abandoned ad-hoc milestones for that parent.

**Risk:** low — requires a specific substring to be typed into a description. Not exploitable for money (the buyer cannot make THEIR OWN ad-hoc abandoned milestones persist past cleanup in a way that benefits them), but makes the query less reliable.

**Suggested fix:** either use MySQL's `JSON_EXTRACT(meta, '$.is_contract_milestone') = TRUE` (if the DB min version allows, MySQL 5.7+/MariaDB 10.2+), or move `is_contract_milestone` out of `meta` and into a dedicated column. The project already has a migration layer that added `contract_type` to the proposals table — adding `is_contract_milestone TINYINT(1) DEFAULT 0` to `wpss_orders` and indexing it would be cleaner and faster.

### M4 — Cascade-cancel listener doesn't emit per-milestone `wpss_milestone_cancelled` / similar hooks
**File:** `src/Core/Plugin.php:796-816`

**What I expected:** when the parent is cancelled, each cascaded milestone should emit a cancellation event so downstream listeners (notification, email, audit log) can react.

**What the code does:** a single bulk `UPDATE` sets all `pending_payment` milestones under the parent to `cancelled`. No per-milestone action is fired. The auto-complete parent listener at `:789-790` fires only on `wpss_milestone_approved` and `wpss_milestone_declined`, not on this cascade — so a parent with some paid+completed milestones and the rest cascade-cancelled from the parent will trigger the auto-complete handler... wait, it doesn't. The cascade happens because the parent was ALREADY cancelled; the auto-complete listener re-reads the parent status and bails because `parent->status === 'cancelled'` (`:765`). OK that bail is correct.

But: no listener gets told about the individual cancelled milestone rows. If the vendor was mid-work on milestone #2, they receive no notification that it was cancelled. They'll next log in and see the status silently flipped.

**Risk:** UX only.

**Suggested fix:** iterate the milestone rows before the bulk update, fire `do_action('wpss_milestone_declined', $ms->id, $parent_order_id, $parent->customer_id)` for each one. The existing `wpss_milestone_declined` listener (line 790) is idempotent — it re-checks status — and the notification/email listeners fire nicely.

## Low findings

### L1 — `ExtensionOrderService::decline` cancel-sub-order `$wpdb->update` conditional uses `status='pending_payment'` guard
**File:** `src/Services/ExtensionOrderService.php:554-567`

**What I expected:** conditional update guarded by status is the right pattern for idempotency under concurrent requests. Code is correct but the function returns `true` even if the `$wpdb->update` affected 0 rows (e.g. the sub-order was already cancelled). The "extension_requests" row still gets flipped to REJECTED at line 543 regardless of whether the sub-order transition succeeded. If the sub-order was already cancelled via another path, the request row flipping to REJECTED is fine.

**Risk:** negligible.

**Suggested fix:** none strictly required; consider checking `$wpdb->update` return value and logging a note if 0 rows were affected when it was expected to affect 1.

### L2 — `MilestoneService::is_locked` calls `get_for_parent()` which re-queries the full milestone list per check
**File:** `src/Services/MilestoneService.php:728-743`

**What I expected:** for a server-side URL-tamper guard on the checkout backstop (called once per pay_order request), the cost is fine.

**What the code does:** each `is_locked($id)` fetches all sibling milestones, computes the lock-step pass, then returns `is_locked` for the requested row. Works correctly but is O(n) per check. Not a problem in practice.

**Risk:** none.

**Suggested fix:** none. Just noting for future. A pointer: if you later add `is_locked` checks to the dashboard list per row, the current implementation would be N² — in that case, compute once and memoize on the service instance.

## Cross-cutting observations

- **Hook listener closure scoping in `define_notification_hooks` is correct.** `$notification_service` and `$email_service` are declared once at the top of the method (Plugin.php:449-453) and every subsequent closure uses `use ($notification_service, $email_service)` — no repeat of the extension-block scoping bug. Verified across the 7 new listeners.
- **Idempotency guards on credit_*_on_payment_complete are correctly keyed** for milestone (`type='milestone', reference_type='order', reference_id=<milestone_id>`) and extension (`type='extension', reference_type='order', reference_id=<pay_order_id>`). Only tip is off (see C1).
- **`decode_order_meta` is correct** and each call site passes a `ServiceOrder` model (from `wpss_get_order`) or the ExtensionOrderService's private `get_order()` (also ServiceOrder). All call sites verified — no raw wpdb rows are fed in.
- **`wpss_order_cancelled` action name is correct** — fired at `src/Services/OrderWorkflowManager.php:742`, listened to at `src/Core/Plugin.php:797`. Cascade-cancel will fire.
- **`wpss_order_paid` has three listeners (TippingService, ExtensionOrderService, MilestoneService) plus OrderWorkflowManager::handle_payment_complete** — each sub-order service gates on its own platform marker so there is no crossfire. The OrderWorkflowManager handler is the one that creates the visible mid-flight `pending_requirements` issue in H3.
- **`InternalWalletProvider::get_balance` SQL debit list (`withdrawal, debit, dispute_refund`) correctly treats every wallet type produced by the codebase.** `order_earning`, `tip`, `extension`, `milestone` are all credits. `order_reversal` writes `amount = -N` and falls into the ELSE branch — `+amount` adds a negative, effectively debiting. `credit` and `debit` types from admin actions are handled. No dispute_refund-typed writer was found in the codebase (only dispute reversals currently use `order_reversal`), so including it in the debit list is forward-looking rather than active.
- **`OrderRepository::get_vendor_stats` placeholder ordering matches** the `$wpdb->prepare` argument list. Count exclusions (tip, extension, milestone) and earnings inclusion (only tip excluded) are consistent with the documented intent. `total_earnings` uses `COALESCE(vendor_earnings, total)` so legacy rows without commission breakdown still contribute.
- **Pro plugin's `sync_wallet_balance_on_commission` listener is removed.** `grep -r wpss_commission_recorded wp-sell-services-pro/` returns no matches. The double-credit source is gone. Pro's `WalletManager::process_vendor_payout` still writes an `order_earning` row on `wpss_order_status_completed` — but CommissionService::record already writes the same row from `handle_order_completed` at priority 10. WalletManager's insert would duplicate it UNLESS WalletManager itself also checks for existing rows. Spot-check: WalletManager at `:340-353` guards on `$order->status==='completed'` and `payment_status!=='wallet_paid'`, but does NOT check for an existing `order_earning` ledger row. If both CommissionService (free) AND WalletManager (pro) are active, each completed order gets TWO `order_earning` wallet rows — a double credit. This warrants its own investigation; it may be intentionally guarded by the Pro not being wired into standalone, but the brief called out exactly this kind of removed-listener regression.
