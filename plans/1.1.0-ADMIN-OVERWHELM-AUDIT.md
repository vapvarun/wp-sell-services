# 1.1.0 Admin Overwhelm Sweep — Daily / Weekly / Monthly cadence audit

> **Goal:** every admin page surfaces information at the right cadence, with bulk-actions where the volume justifies them, so a single admin scaling to a busy marketplace doesn't drown.
>
> **Methodology:** browser walk every WPSS admin page as `varundubey` (administrator), categorise each surface by the cadence at which an admin should engage with it, then compare the rendered UX against that cadence's needs. Companion to `1.1.0-COMPLETENESS-AUDIT.md`.
>
> **Date:** 2026-04-26.

---

## Cadence model

| Cadence | What admin should see | Fix bar |
|---|---|---|
| **Daily** | Disputes needing mediation, vendor escalations, high-value withdrawal requests, new orders that need manual intervention. | A single screen surfaces all daily action items with counts. Per-row + bulk actions. |
| **Weekly** | Moderation queue, new vendor approvals (in approval mode), commission rate review, refund / cancellation patterns. | Filter + sort + bulk on each. |
| **Monthly** | Settings tuning, top-line marketplace metrics, taxonomy maintenance. | Stable-state reads — no triage needed. |
| **Never (auto)** | Routine workflow that should never reach admin. | If admin sees it, that's the friction. |

## Findings (severity-tagged)

### 🔴 Broken — fix for 1.1.0

1. **Orders admin: "Service" column shows literal text `Deleted`** for orders whose `service_id` points to a removed CPT row.
   * Reproduced: 11 of 14 orders in the test DB display the `Deleted` label instead of the original service title.
   * Same regression visible on the Dashboard "Recent Orders" widget.
   * Fix direction: store `service_title` snapshot at order-creation time (it's already in `wpss_orders.meta` for some flows) and fall back to that for deleted services, with a discreet "(deleted)" suffix so admin still has context. Failing that, show the order_number + customer + total without a misleading "Deleted" placeholder.

### 🟠 Incomplete — should fix for 1.1.0

2. **Withdrawals admin lacks bulk actions** — Daily/Weekly-cadence page where admin most needs scale.
   * Currently: per-row "Approve" + "Reject" only. With 50 pending withdrawals on a busy day, that's 50 individual clicks.
   * Fix direction: add a checkbox column + a `Bulk Actions` dropdown with `Approve`, `Reject`, `Mark as paid` (matches the same affordance Service Moderation already ships).

3. **Vendors admin lacks bulk actions** — for sites running `vendor_registration = 'approval'`, batch-approving the weekly inflow is the entire point of that mode.
   * Currently: per-row "View / Edit User / Suspend" only.
   * Fix direction: checkbox column + `Bulk Actions` with `Approve` (turns Pending → Active), `Suspend`, `Reactivate`. Mirrors Service Moderation's approve/reject pattern.

4. **Dashboard misses Daily-cadence KPIs** — the four headline tiles are revenue / volume metrics; they don't tell the admin **what's on their plate today**.
   * Currently surfaced: Total Orders, In Progress, Completed, Total Revenue, plus a `Pending Orders` link in the Content Overview list.
   * Missing as top-line tiles: `Open Disputes`, `Pending Withdrawals` (count + summed amount), `Pending Vendor Approvals` (only relevant in approval mode).
   * Fix direction: split the dashboard into two strips — "Action Items" (Daily) on top, "Marketplace Health" (rolling totals) below. Action Items tiles link directly to their respective pages with counts ≥ 1; tiles with count = 0 dim out so they fade visually but stay present so admin can confirm "nothing on my plate today".

### 🟡 Polish — defer to 1.1.x if scope balloons

5. **Withdrawals: no date-range filter, no amount sort.** With volume, high-value-first sorting helps direct attention.

6. **Vendors: no date-range filter** (joined-date). Easy add given the column already exists.

7. **Orders: status tabs miss workflow states** — `pending_requirements`, `requirements_submitted`, `pending_approval`, and `disputed` are all real lifecycle statuses but absent from the tab strip. Admin can't quickly filter to the requirements-blocked queue.

8. **Disputes: filters not testable** in the empty state, but on a populated table the same per-status / by-age / by-reason cuts that Withdrawals + Orders need will be needed here too — pre-build them so the page doesn't ship as a flat list.

## Out-of-scope for the sweep itself

* Bug fixes for issues found above — those land on `feat/1.1.0-completeness` (combined with the matrix audit's findings) so the release ships one merge.
* Pro plugin admin pages (Razorpay, Stripe Connect, Subscriptions, etc.) — separate sweep when those are fully populated with test data.
* WP-list-table pagination / per-page screen options — already present via WP core defaults.
