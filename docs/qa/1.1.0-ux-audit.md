## Critical findings

### F1 — Lock-step on the buyer order page is invisible; every pending phase shows an enabled Pay button

**Journey:** B, C
**Step:** B6 — buyer lands on a new milestone order; the parent has phase 1, 2, 3 all pre-created as `pending_payment`.
**What user expects:** "Only phase 1 should be payable right now; phases 2 and 3 should show as locked with a reason."
**What actually happens:** The server-side gate exists (`MilestoneService::is_locked()` is correctly enforced in the checkout handler, per QA 7d.7 / 14.5), but the visible order-view template (`templates/order/order-view.php` lines 1229–1332) iterates every milestone and renders the **Accept & Pay %s** button for *any* phase whose status is `pending_payment`, with no branch on `$m['is_locked']`. The state label says "Awaiting your payment" for all three phases. The buyer has no way to know phase 2 and 3 aren't really available — clicking Pay on phase 2 takes them to checkout and then dies on a locked-milestone error from the server.
**Why it confuses:** The buyer either (a) believes they're about to prepay the whole contract and panics, or (b) clicks phase 2 trying to be efficient and gets a generic "locked" server error mid-checkout. Either way it destroys trust in the flow on the very first interaction. The QA checklist explicitly expects "Phases 2 & 3 show locked state (no Pay button, 'Locked · pay phase N first' or similar)" — the backend supports it, the UI doesn't render it.
**Suggested fix:** In the milestone loop, branch on `$m['is_locked']`:
- Replace the Pay button with a disabled pill `Locked — pay phase N first` (where N is the previous non-terminal phase's number).
- Change state label to `Locked · unlocks after phase N`.
- Add a short lead paragraph above the timeline on new milestone orders: *"You pay each phase when you're ready. We'll unlock the next phase after the current one is approved."* — so the buyer understands the rhythm without trial-and-error.

### F2 — Buyer accept-proposal confirm is generic; buyer doesn't know they're signing up for a multi-phase lock-step contract

**Journey:** B
**Step:** B4 — buyer clicks Accept Proposal on a milestone-based proposal.
**What user expects:** "Before I lock in a 3-phase / $500 contract, I want to see a confirmation that says *exactly* that, with the phases and the rule spelled out."
**What actually happens:** `assets/js/frontend.js` line 1104 uses the same generic prompt for fixed and milestone proposals: *"Accept this proposal and create an order?"*. No phase count, no total, no mention of the lock-step payment model, no mention that phases are pre-created and will sit in their inbox. After click, the parent order is created, the buyer lands on it with total=$0, `status=in_progress`, `payment_status=paid` — state that's shocking if they expected a checkout step.
**Why it confuses:** The buyer has the strongest impression of what they're agreeing to AT the moment of click. A cheap one-liner confirm on a Fiverr-vs-Upwork-grade distinction is malpractice. QA 7c.3 even calls it out: *"Buyer NOT taken to a checkout page; No pay_order step for the parent — straight to order view."* This is the correct behavior but also the *least* expected one — the user needs to be warned.
**Suggested fix:** When the proposal being accepted has `contract_type === 'milestone'`, replace the confirm with a modal that shows:
- "You're accepting a **3-phase milestone contract** with {Vendor}."
- Phase list (title + amount) as a mini-summary.
- Total: $500.
- "You pay nothing now. You'll pay each phase as you're ready — we unlock the next phase after you approve the current one."
- `[Cancel]` / `[Accept & Start Project]` buttons.

The data is already on the page (the breakdown `<details>` has it); reuse it.

### F3 — Parent order auto-completion has no celebratory / summary state; project just silently flips

**Journey:** C
**Step:** C8 — after phase 3 is approved, the `auto_complete_parent` closure in `src/Core/Plugin.php` flips the parent to `completed`, sets `completed_at`, and writes nothing else.
**What user expects:** "Yay, we finished the project! Give me a summary: total paid, vendor earnings summary, 'Leave a review' CTA, maybe a download-all-deliverables option."
**What actually happens:** The parent order's status badge changes from `In Progress` to `Completed` on the next page load. No in-app notification is dispatched (the listener only writes DB), no email template is fired for "project wrapped", no review CTA change on a milestone-contract parent. The QA checklist covers status-mechanics but no UX wrap-up. On completion, the Tip CTA does appear (lines 1604-1668 render when `order->status === 'completed'`), but it's the only celebratory cue and it sits at the bottom of a page whose main content (timeline) still looks like an in-progress project.
**Why it confuses:** Users close browser tabs looking for signal that they're done. There is none. Both buyer and vendor are left unsure whether to keep watching this order or move on. Vendors especially, since the last approval happened on the milestone sub-order view (not the parent), may never revisit the parent at all.
**Suggested fix:**
- Render a *Project Completed* card at the top of the parent's timeline once every phase is terminal: headline + date completed + total phases paid + total vendor earnings (vendor view) / total paid (buyer view) + Review button (buyer) + "Done" stamp on each phase.
- Fire a new `milestone_project_completed` notification/email for both parties ("Your project with X is complete.").
- Optionally offer a "Download all deliverables" link that bundles every phase's delivery attachments.

## High findings

### F4 — Acceptance of a milestone proposal with ZERO phases or ONE phase is silent about contract structure

**Journey:** A, B
**Step:** A4/A5 — vendor adds zero rows or a single row and submits.
**What user expects:** "If I'm only doing one phase, am I in Fixed mode by accident? Can I still do Milestone-based with one phase?"
**What actually happens:** JS validation (frontend.js 1158-1172) requires `>= 1` milestone row with title + amount > 0, so a zero-row submit is blocked with `"Add at least one milestone phase."`. A 1-row milestone submits fine. On the buyer's side, the badge says `1 phase milestone plan`, which is technically correct but reads like a Fixed proposal with extra steps.
**Why it confuses:** Vendors don't know whether a 1-phase milestone plan has any advantage over Fixed — the two look identical on outcome (one payment, one delivery). For a buyer seeing both options side-by-side, the "1 phase milestone" looks like a gimmick.
**Suggested fix:** In the proposal form, when a vendor saves with exactly one milestone row, show an inline hint: *"A single-phase milestone acts like a fixed-price contract. Consider switching to Fixed price if that's what you meant."* Don't block — just nudge.

### F5 — Buyer has no native "Request revision in chat" affordance on milestone delivery

**Journey:** C5
**Step:** Buyer is on `milestone-view.php` reviewing a `pending_approval` phase.
**What user expects:** "Approve or Request Revision — both as big clear buttons." (This is the shape of the parent-order revision flow lines 239-245.)
**What actually happens:** The milestone sub-order view shows only `[Approve delivery]`. The submit-delivery modal's intro text says *"They will review and approve, or ask for changes in chat"* — but there is no **Ask for changes in chat** button. The buyer has to: go back to parent order → find conversation thread → type their request → manually reference phase N → hope vendor re-submits. That's 4+ clicks and a mental translation step for what on the parent order is a single button.
**Why it confuses:** Buyers have *opposite* mental models between parent-order (catalog) and milestone (request) flows. On a catalog order, revision has a first-class button with revision counter and a dedicated modal. On a milestone phase, revision is only mentioned in copy somewhere else. Users who've used the marketplace for catalog work will assume the button exists and will either accept prematurely (because they can't find revision) or open a dispute.
**Suggested fix:** Add a secondary `[Request changes]` button next to `[Approve delivery]` that opens a simple modal with a textarea, posts a chat message to the parent order's thread, and flips the milestone's status back to `in_progress` (or a new `milestone_revision_requested` state). This matches the parent-order revision flow users already know.

### F6 — Invalid intro-video URL clears the field silently; vendor has no idea it was rejected

**Journey:** F3
**Step:** Vendor pastes a Loom / Wistia / Dailymotion URL into the Intro Video field and hits Save.
**What user expects:** "If my URL isn't supported, tell me that in the form so I can go grab a YouTube link instead."
**What actually happens:** `src/Frontend/AjaxHandlers.php` line 3613-3614 runs `wpss_is_supported_video_url()`; on fail it **sets the stored value to an empty string**. The form saves "successfully", the page reloads, and the input is now empty. No error message, no "Only YouTube / Vimeo supported" validation error, no warning.
**Why it confuses:** The vendor assumes their video uploaded fine until they later visit their own profile and notice no video. Then they try again, same result, and conclude "the feature is broken." The hint below the field mentions YouTube/Vimeo, but most people don't re-read hints after the first save.
**Suggested fix:**
- Either: make the AJAX handler return a non-fatal warning (`unsupported_provider`) and surface it as a toast `"We only support YouTube and Vimeo URLs. Other providers were not saved."`.
- Or: client-side validate on blur — if `url.match(/youtube\.com|youtu\.be|vimeo\.com/)` fails, show inline error and prevent submit.

### F7 — Mid-contract scope-change pattern is undiscoverable to the buyer

**Journey:** D (buyer wants to add scope)
**Step:** Buyer is mid-project (phase 2 `in_progress`), realises they want extra screens.
**What user expects:** "Some way to say *hey, we need to add X* without opening a dispute." On Upwork there's a clear *Request additional milestone* flow.
**What actually happens:** Nothing exists on the buyer side. The only mechanism is the VENDOR clicking `+ Propose Milestone`. A buyer who doesn't already know this has zero affordance to push the pattern — they'll either open a dispute, ask in chat and wait, or order a new parent service.
**Why it confuses:** The whole selling point of the milestone model over fixed-price is scope flexibility. Hiding that flexibility behind a vendor-only button defeats the feature.
**Suggested fix:** Add a secondary "Need more work done? Ask seller for a new phase." link/button on the buyer's milestone timeline that opens a chat-prefill modal: *"Describe the extra work you need; we'll ping your seller to propose a phase for it."* The message lands in the conversation thread with a consistent format so the vendor knows exactly what to quote.

### F8 — Cascade-cancel only covers `pending_payment` milestones; `in_progress` (paid, not delivered) phases are abandoned with no UX explanation of the money position

**Journey:** D5
**Step:** Parent order is cancelled while phase 2 is `in_progress` (buyer paid, vendor hasn't submitted).
**What user expects:** "Tell me clearly what happens to the money on phase 2: refunded, held, released?"
**What actually happens:** `src/Core/Plugin.php` lines 796-814 UPDATE only `status='pending_payment'` rows to `cancelled`. An `in_progress` milestone stays `in_progress` even after parent cancellation. The vendor's wallet already has the NET from phase 2 (money settled on payment). There's no UI surface telling buyer OR vendor what the state of phase 2 means now — the parent is cancelled but the phase hangs unresolved.
**Why it confuses:** This is the highest-stakes moment in the flow (real money already moved), and there is no copy. Buyer assumes "cancelled = refunded" and disputes. Vendor assumes "I was paid, I should still deliver" and keeps working. Customer support fields 12 tickets.
**Suggested fix:**
- Explicit policy copy in the cancel modal when there's a paid-but-not-delivered phase: *"Phase N was paid and is in progress. Cancelling the project does NOT automatically refund this phase. You'll need to open a dispute or agree with your seller on a partial refund."*
- On the vendor side, when they load a parent that was cancelled-while-in-flight, show a banner: *"This project was cancelled by the buyer. Phase N was already paid — continue delivery or agree on a partial refund via chat/dispute."*
- Consider auto-setting `in_progress` phases to a new `parent_cancelled_review` state that queues for admin/dispute mediation.

### F9 — Vendor sees no email nudge or dashboard surface when a delivered milestone sits unreviewed for days

**Journey:** D3 (buyer ignores submitted milestone for 30 days)
**Step:** Vendor submitted phase 1, buyer never approves, time passes.
**What user expects:** "I'm owed approval; the system should auto-complete or at least remind the buyer."
**What actually happens:** The generic `auto_complete_orders` cron (OrderWorkflowManager line 243-292) runs for service orders — it triggers on `pending_approval` + `deliver_at` age, and flips to `completed`. It's unclear whether milestone sub-orders are also swept (code path exists but the cron's status target set is for main orders). Either way: money is already in vendor wallet from the payment step, so no urgency on the vendor's ledger. But for the *flow*, a milestone hanging 30 days in `pending_approval`:
- Blocks the next phase from unlocking.
- Blocks the parent from auto-completing.
- Leaves the Review CTA dangling for the buyer with no nag.
There is no dashboard indicator "awaiting 21 days — remind buyer" for the vendor, and no auto-reminder email to the buyer.
**Why it confuses:** Vendor thinks the platform is broken. Buyer forgets entirely. Both parties lose interest. The project slowly dies.
**Suggested fix:**
- Scheduled email to buyer after N days in `pending_approval` with a subject like *"Phase X from {Vendor} is waiting for your review"* and a direct link to `milestone-view.php` with the Approve button.
- Dashboard badge on the vendor's parent-order card when a phase sits unreviewed past a threshold: `Awaiting buyer review · 14 days`.
- Honour the existing `auto_complete_days` setting on milestone sub-orders (verify the cron sweeps them) or extend the cron to include milestone platform.

**Journey:** D4
**Step:** Vendor clicks `Cancel proposal` on a pending_payment milestone they proposed.

## Cross-cutting observations

- **Lock-step is the most important distinguishing feature of the Upwork-style flow, and it is enforced server-side but invisible client-side (F1). This is the single biggest perception gap in the release.**
- **Acceptance commits the buyer to a 3-phase lock-step contract via a generic "Accept this proposal?" confirm (F2). Every other first-run Fiverr/Upwork UX I've seen double-gates this because of how sticky and public the consequence is.**
- **The closing moments of the flow are silent — no celebratory/summary state on project auto-completion (F3). Users don't know when they've finished.**
